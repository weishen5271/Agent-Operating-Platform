## 1. 文档目标与读者

### 1.1 目标

本文回答四个问题：

1. 不同行业有哪些值得 Agent 赋能的高价值场景？
2. 想把一个行业接进 Agent Operating Platform，要交付什么、按什么流程？
3. 平台层是怎么把这些"行业差异"装配进同一个底座的？
4. 怎么保证扩展行业不污染平台核心代码与数据库？

### 1.2 读者

| 角色 | 关注点 |
|------|------|
| 行业方案架构师 | 场景识别、业务包边界划分、可行性评估 |
| 业务包开发负责人 | 交付物清单、开发流程、对接路径 |
| 平台运行时维护者 | 扩展点、装配流程、治理一致性 |
| 行业客户 / 业务方 | 能用 Agent 做什么、如何参与共建 |

### 1.3 与制造业业务包的关系

[制造业设备运维助手](制造业设备运维助手-业务包开发文档.md) 是首个完整业务包样板。本文是从中提炼出的**通用开发方案**，供金融、医疗、政务、零售、法务等其他行业参考；同时把"行业接入应该走的路径"沉淀为平台规范。

## 2. 什么是行业业务包

### 2.1 定义

业务包（business_package）是平台调度的**装配单元**，把一个行业的"高频场景剧本 + 必须遵守的合规边界"显式编码下来，让 Agent 在边界内自动协调多个系统、多种知识、多类角色完成业务闭环。

业务包不是：

- ❌ 一组散装插件
- ❌ 一份提示词模板
- ❌ 一个微调模型

业务包是：

- ✅ 场景剧本 + Capability 组合 + 知识源绑定 + 角色与审批策略 + 评估集 + 灰度上线方案的**集合体**

### 2.2 与平台底座的边界

```text
平台底座（不感知行业）
  对话编排 / Planner / Retriever / Memory / Risk / Audit / 模型路由 / 配额 / 灰度
        ↑ 通过 yaml 与插件接入，不需要改底座代码
行业业务包
  场景剧本 / Capability 组合 / 知识 / 提示词 / 阈值 / 评估
```

平台只提供"治理与编排能力"，行业逻辑下沉到业务包。新增一个行业 = 新增一个 `packages/<industry>/` 目录 + 若干插件，**不改平台核心表、不改 Runtime 核心代码**。

## 3. 常见行业的 AI 赋能场景

下列是经过验证、值得做成业务包的高价值场景，按行业归类。每条都按"用户角色 / 痛点 / 业务包能闭环什么"组织。

### 3.1 金融（银行 / 保险 / 证券）

| 业务包候选 | 用户角色 | 痛点 | 业务包闭环 |
|------|------|------|------|
| `fin_credit_assistant` 信贷助手 | 客户经理 | 资料散在多个系统、合规材料清单变动频繁 | 自动核对资料齐套、预审风险、生成尽调摘要、起草上会材料草稿 |
| `fin_kyc_aml` 反洗钱与 KYC | 合规专员 | 工单多、规则更新快、误判率高 | 关联交易聚合、可疑模式提示、SAR 报告草稿、规则版本可追溯 |
| `fin_claim_assistant` 保险理赔 | 理赔员 | 证据材料杂、规则细 | 单证 OCR 与抽取、责任划分建议、赔付方案草稿、客户问询自动回复草稿 |
| `fin_advisor` 财富顾问助理 | 理财经理 | 产品多、客户画像更新慢 | 客户适配产品筛选、风险等级核对、合规话术、营销材料草稿 |

**赋能特征**：高合规、强审批、双重审计、敏感信息脱敏、不允许模型自主下单。

### 3.2 医疗与健康

| 业务包 | 用户角色 | 痛点 | 闭环 |
|------|------|------|------|
| `med_clinical_assistant` 临床决策辅助 | 主治医师 | 病史散、指南更新快 | 病历摘要、用药交互提示、检查建议草稿（必须医师签字） |
| `med_nursing_handover` 护理交接 | 护士长 | 交接班信息易丢失 | 自动生成交班单、异常患者高亮、异动追踪 |
| `med_quality_audit` 病案质控 | 病案室 | 质控规则多、人工查不过来 | 病历完整性检查、编码建议、退回清单 |
| `health_triage` 互联网医院导诊 | 平台运营 | 分诊不准、转科多 | 症状分诊、科室推荐、就诊须知、避免误导式诊断 |

**赋能特征**：所有写操作必须有医师/护士签字；引用必须可追溯到指南版本；严格不输出诊断结论，只输出辅助信息。

### 3.3 政务与公共服务

| 业务包 | 用户 | 痛点 | 闭环 |
|------|------|------|------|
| `gov_one_window` 一窗通办 | 综合窗口工作人员 | 事项多、材料清单复杂 | 按事项匹配材料清单、缺件提醒、办件草稿、跨部门流转提示 |
| `gov_12345` 12345 工单 | 话务员 / 派单员 | 派单错误率高、回访不及时 | 工单分类、自动派单建议、超期预警、回访话术 |
| `gov_policy_qa` 政策问答 | 公众 / 企业服务专员 | 政策多、版本更新快 | 严格引用最新政策原文、避免编造、留痕可审 |
| `gov_inspection` 执法检查 | 执法人员 | 现场检查清单繁琐 | 按行业生成检查清单、违法证据收集模板、文书草稿 |

**赋能特征**：政策原文强引用、可解释、留痕、避免任何"自由发挥"；版本治理是核心。

### 3.4 法务与合同

| 业务包 | 用户 | 痛点 | 闭环 |
|------|------|------|------|

**赋能特征**：引用法律法规具体条款；不输出法律意见结论，只输出"备审材料"。

### 3.5 零售与电商

| 业务包 | 用户 | 痛点 | 闭环 |
|------|------|------|------|
| `retail_store_ops` 门店运营 | 店长 | 多系统切换 | 销售日报、库存预警、补货建议、人员排班调整 |
| `retail_after_sales` 售后客服 | 客服 | 工单堆积、政策记不清 | 工单分类、政策匹配、退换货草稿、客户回复草稿 |

## 4. 业务包通用结构

> **v1.1 实施版（2026-04-27）**：业务包以**可上传 bundle**形式交付，
> 通过 **业务包管理 → 导入业务包** 选择 zip 上传，后端落到
> `packages/installed/<package_id>/` 并自动注册私有 skill / tool / plugin。
> 不再要求开发者把 manifest 复制到 `packages/catalog/`（catalog 仅保留平台样例）。
> manifest 用 JSON（与运行时 PackageLoader 对齐）；YAML 形态作为编辑期可读副本，编译为 JSON 后入 bundle。

任何行业的业务包都遵循同一组成结构：

```text
<package_id>/                            # zip 顶层目录
├── manifest.json                        # 元数据 + requires + provides 索引（必需）
├── prompts/
│   ├── system_prompt.txt                # 行业行为准则
│   └── planner_prompt.txt               # 剧本规划提示词
├── skills/                              # 业务包私有 skill（声明式 JSON）
│   └── <skill>.json
├── tools/                               # 业务包私有 tool 别名（可选）
│   └── <tool>.json
├── plugins/                             # 业务包私有 plugin（声明式 stub，第一阶段不接受代码）
│   └── <plugin_name>/plugin.json
├── knowledge-bindings.yaml              # 知识源绑定 + 扩展属性 schema + 检索过滤模板
├── approval_policy.yaml                 # 审批阈值与角色
├── evals/
│   ├── golden_qa.yaml                   # 标准问答集
│   ├── action_set.yaml                  # 写操作样本
│   ├── redline.yaml                     # 红线对抗集
│   └── skill_<name>.yaml                # skill 级评估集
└── README.md
```

### 4.1 manifest.json 关键字段

```jsonc
{
  "package_id": "industry.<name>",
  "name": "...", "version": "v1.0.0", "owner": "...", "status": "灰度中", "domain": "industry",
  "intents": [...], "knowledge_bindings": [...],

  // 平台已存在的依赖（兼容性校验，不安装任何东西）
  "requires": [
    { "kind": "common_package",  "name": "_common/knowledge",  "version_range": ">=1.0 <2.0" },
    { "kind": "platform_skill",  "name": "kb_grounded_qa",     "version_range": ">=1.2 <2.0" }
  ],

  // 业务包私有产物 — 上传 bundle 时一并安装
  "provides": {
    "skills":  ["skills/<skill>.json"],
    "tools":   ["tools/<tool>.json"],
    "plugins": ["plugins/<plugin_name>/plugin.json"]
  },

  "prompts": { "system": "prompts/system_prompt.txt", "planner": "prompts/planner_prompt.txt" }
}
```

私有 skill / tool / plugin 文件由 PackageLoader 自动解析，注入 SkillRegistry / ToolRegistry /
PluginRegistry 时打上 `source="package"` + `package_id=<本包>` 命名空间，不会污染全局。

### 4.2 上传与卸载 API

| 操作 | 接口 |
|------|------|
| 安装 | `POST /api/v1/admin/packages/import`（multipart `file=<zip>`，可选 `?overwrite=true`） |
| 卸载 | `DELETE /api/v1/admin/packages/<package_id>/bundle` |
| 列表 | `GET  /api/v1/admin/packages` |

约束：
- zip 大小 ≤ 50 MB、文件数 ≤ 500、仅允许 `.json/.yaml/.yml/.txt/.md` 扩展名
- 路径不允许 `..` 越界、不允许绝对路径

### 4.3 plugin.json 中的 `executor` 类型

| executor | 谁提供执行逻辑 | bundle 需带什么 | 适用 |
|---|---|---|---|
| `stub` (默认) | 平台 `StubPackagePlugin` | 仅契约；自动按 `output_schema.required` 合成 fixture | 上传后立即可调，等待真实接入 |
| **`http`** | 平台 `HttpExecutor`（一次性写） | 每个 capability 自带 `binding` 块（method/path/headers/body/response_map/error_translation） | 90% REST 系统（CMMS / ERP / SCADA REST 等） |
| `mcp` *(规划中)* | 平台 `McpExecutor` | `binding.mcp_tool` + `argument_map` | 已 MCP 化的系统 |
| `platform` *(规划中)* | 平台预装 Python 插件包 | bundle 仅写 `platform_plugin: "<name>@<version>"` | 复杂协议、特殊鉴权、需要状态机 |

### 4.4 HTTP binding DSL

`plugin.json → capabilities[].binding` 支持以下引用语法（任何字符串值里）：

| 引用 | 含义 | 来源 |
|---|---|---|
| `$input.<dot.path>` | 调用 capability 时的 payload | Planner / Skill 步骤 |
| `$config.<dot.path>` | 租户级 plugin_config | `PUT /admin/plugins/<name>/config` 写入 |
| `$secret.<name>` | 凭据，等同 `$config.secrets.<name>` | plugin_config 里的 `secrets` 子键 |
| `$response.<dot.path>` | HTTP 响应（仅 `response_map` / `error_translation` 中可用） | 调用后注入 |

binding 完整示例（来自 example/cmms_work_order）：

```jsonc
{
  "method": "GET",
  "path":   "/api/v1/workorders",
  "query":  { "equipment_id": "$input.equipment_id", "limit": "$input.last_n" },
  "headers":{ "Authorization": "Bearer $secret.cmms_token" },
  "body":   { /* 仅 POST/PUT/PATCH */ },
  "timeout_ms": "$config.timeout_ms",
  "response_map": {
    "workorders": "$response.body.data",
    "total":      "$response.body.total"
  },
  "error_translation": {
    "401": "CMMS_AUTH_FAILED",
    "404": "CMMS_NOT_FOUND",
    "5xx": "CMMS_UPSTREAM_UNAVAILABLE"
  }
}
```

### 4.5 安全边界

binding DSL 是**纯数据**，不是代码：

- 不支持循环 / 函数 / 变量赋值；只能解析上述 4 类引用
- `endpoint` 必须由运维在 plugin_config 中显式填入；bundle 不能写死外部 URL
- `secret` 永远从 plugin_config.secrets 读取，bundle 见不到明文
- 写操作 (`side_effect_level=draft/write`) 仍走 confirm_draft / 审批，与 stub 时代一致
- 响应大小 1MB 上限，避免大 body 拖垮链路

并通过若干插件包提供 capability：

```text
plugins/<industry>_<system>/    # 如 fin_core / med_emr / gov_12345
├── plugin.yaml
├── capabilities/*.json
├── src/
│   ├── handlers.py
│   ├── mapping.py
│   └── client.py
└── tests/
```

业务包 = 配置 + 提示词 + 评估，**不写业务编排代码**；插件 = 与外部系统交互的通道。两者解耦。

## 5. Capability / Tool / Skill 编排

业务包在剧本里要调度的不止 capability。平台支持三类"动作单位"，分别承担不同抽象层级的工作。本章说明它们的边界、如何在业务包里声明，以及编排原则。

### 5.1 三者的定位与边界

```text
Capability (插件能力)   ─ 与外部系统对接的原子能力，强契约、强治理
Tool (工具)             ─ 通用、无状态、跨业务复用的小工具
Skill (技能)            ─ 多步组合的可复用剧本片段，封装一段业务推理
```

| 维度 | Capability | Tool | Skill |
|------|------|------|------|
| 来源 | 插件包提供 | 平台内置 / 通用包 | 业务包自带 / 跨业务共享 |
| 调外部系统 | ✅ 主要工作 | ❌ 一般不调 | 通过 capability 间接调 |
| 状态 / 副作用 | 可写、必须幂等 | 通常无副作用 | 取决于内部 capability |
| 强契约 schema | 强制 | 强制 | 输入输出必须可序列化 |
| 治理 | PDP + 审批 + 审计 | 限流 + 审计 | 走内部 capability 的治理 |
| 示例 | `erp.spare.stock.query` | `text.summarize`、`time.diff`、`math.calc` | `fault_triage`、`spare_lookup_with_alt` |

**判别口诀**：

- **Capability** = "我去 X 系统办一件事"
- **Tool** = "我做一次纯计算 / 格式化 / 文本处理"
- **Skill** = "我按这个剧本组合好几步，封装成一个动作"

### 5.2 平台内置 Tool 速查

| Tool | 用途 |
|------|------|
| `text.summarize` | 文本摘要 |
| `text.translate` | 翻译 |
| `text.extract` | 按 JSON Schema 做结构化抽取 |
| `time.now` / `time.parse` / `time.diff` | 时间处理 |
| `math.calc` | 安全表达式计算 |
| `unit.convert` | 单位换算 |
| `code.run_sandbox` | 受限沙箱代码执行 |
| `web.fetch` | 走白名单的网页抓取 |
| `image.ocr` / `image.describe` | 图像识别 |

Tool 默认 `risk_level: low`、不走审批，但仍受配额、Trace、安全扫描覆盖。**Tool 不得被用于绕过 capability 的治理**——例如不要用 `web.fetch` 去爬本应走 capability 的内部系统。

### 5.3 在业务包里声明 Tool 白名单

`capability-map.yaml`：

```yaml
include:                       # capability 白名单
  - erp.spare.stock.query
  - knowledge.search
  - eam.workorder.draft.create

tools:                          # tool 白名单
  - text.summarize
  - text.extract
  - time.diff
  - math.calc
```

Tool 不在白名单里就不能用。Planner 看到的候选池受白名单约束。

### 5.4 Skill：业务包私有的剧本片段

当某段动作序列在多个剧本里重复出现，就抽成 Skill。Skill 不是新插件、也不是新 capability，它是一段"按数据流串好的固定步骤"，可以被剧本一行调用。

#### 5.4.1 Skill 文件结构

```text
packages/<industry>/skills/
├── <skill_a>.yaml
└── <skill_b>.yaml
```

#### 5.4.2 Skill 定义示例

```yaml
# skills/fault_triage.yaml
skill:
  name: fault_triage
  version: 1.0.0
  description: 根据设备号与故障码，输出故障语义、安全等级、推荐处置摘要
  inputs:
    equipment_id: { type: string, required: true }
    fault_code:   { type: string, required: true }
  outputs:
    fault_meaning:     { type: string }
    safety_critical:   { type: boolean }
    suggested_actions: { type: array }
    related_history:   { type: array }

steps:
  - id: profile
    capability: eam.equipment.profile.query
    input:
      equipment_id: $inputs.equipment_id

  - id: kb
    capability: knowledge.search
    input:
      query: "故障码 ${inputs.fault_code} ${profile.equipment_model}"
      filters:
        fault_code_tags: [$inputs.fault_code]

  - id: history
    capability: eam.workorder.history.query
    input:
      equipment_id: $inputs.equipment_id
      fault_code: $inputs.fault_code
      last_n: 5

  - id: extract
    tool: text.extract
    input:
      schema: fault_summary_schema
      source: [$kb.matches, $history.workorders]

outputs_mapping:
  fault_meaning:     $extract.meaning
  safety_critical:   $kb.safety_critical
  suggested_actions: $extract.actions
  related_history:   $history.workorders
```

#### 5.4.3 Skill 的强约束

- 只能写"步骤 + 数据流引用"，**不允许**写自由代码逻辑（否则就退化成插件）
- Skill 自身没有副作用，副作用都来自其内部 capability
- Skill 只接受可序列化的输入输出，便于 Trace 留痕和评估
- Skill 默认私有于本业务包；跨业务包共享需走"通用 skill 包"机制并经审核

### 5.5 在剧本里编排三种动作

`capability-map.yaml`：

```yaml
intent_to_capabilities:
  fault_handling:
    sequence:
      - skill: fault_triage              # 一行替代多步
        input:
          equipment_id: $context.equipment_id
          fault_code: $message.fault_code
      - erp.spare.stock.query
      - shift.skill.match
      - tool: text.extract               # 工具直接混在序列里
        params:
          schema: dispatch_summary_schema
          source: $previous
      - eam.workorder.draft.create
      - erp.spare.requisition.draft
      - shift.dispatch.draft
```

### 5.6 Planner 的优先级

Planner 在业务包上下文中，候选池 = `capabilities ∪ tools ∪ skills`。`prompts/planner_prompt.txt` 里建议加一段：

```text
执行动作时优先使用 skill；同一目标存在 skill 时不要拆成裸 capability 序列。
仅当 skill 不覆盖你的目标时才直接调 capability。
工具(tool)只用于纯文本/计算任务，不要用工具替代外部系统调用。
```

优先级口诀：**Skill 优先 → Capability 兜底 → Tool 辅助 → 自由生成最后**。

### 5.7 与外部 MCP Tool / Server 的接入

外部 MCP 生态（如 GitHub MCP、Slack MCP）按平台架构 §8.6 双向映射规则接入：

| 外部 MCP Tool 类型 | 接入方式 |
|------|------|
| 无副作用、纯查询 | 注册为平台 **Tool** |
| 有写操作、有副作用 | 必须包成 **Capability**，跑完整治理 |

**强约束**：

- 业务包不能直接连任意 MCP Server，必须通过平台注册的插件
- 外部 MCP Server 由插件包封装为 client，仍遵循插件规范的密钥与租户配置约束
- 写操作 MCP Tool 注册为 Tool 一律拒绝

### 5.8 三者的编排策略对照

| 场景 | Capability | Tool | Skill |
|------|------|------|------|
| 查 ERP 库存 | ✅ | ❌ | ❌ |
| 把工单列表摘要成一段话 | ❌ | ✅ `text.summarize` | ❌ |
| "故障分诊"在多剧本里重复 | ❌ | ❌ | ✅ |
| 计算两个时间点之间的工时 | ❌ | ✅ `time.diff` | ❌ |
| 起草工单（写操作） | ✅ | ❌ | ❌ |
| 备件 + 替代件 + 库存的组合查询 | 单独 ❌ | ❌ | ✅ |
| 调外部 GitHub MCP 写 PR | ✅（包成 capability） | ❌ | ❌ |

### 5.9 治理与评估要求

1. **Trace**：Skill 调用产生父 step，内部 capability 是子 step；审计台支持折叠/展开
2. **评估**：抽出来的 Skill 必须有专属评估集（`evals/skill_<name>.yaml`），避免改动后悄无声息影响多个剧本
3. **版本治理**：Skill 升 minor/major 规则与 capability 一致；业务包 yaml 引用 skill 时写版本范围
4. **私有/共享**：默认私有；共享要走"通用 skill 包"机制
5. **Tool 输入输出**：必须可被 Trace 序列化，不允许 Tool 把大对象/二进制塞进结果

### 5.10 一句话总结

- 业务包**编排** Capability / Tool / Skill 三类动作
- Capability 来自插件，Tool 来自平台或通用包，Skill 由业务包自己定义
- 编排原则：**Skill 优先 → Capability 兜底 → Tool 辅助**
- 三者治理一致：写操作必须走 capability，Skill 不能逃避审批，Tool 不能绕过 capability

## 6. 对接平台的整体流程

新行业从 0 到上线的标准路径，分六步：

### 6.1 步骤 1：场景识别与可行性评估（1-2 周）

业务包负责人与行业客户共同输出：

- 高频场景清单（≥ 5 条），每条注明频次、当前耗时、人员痛点
- 每个场景需要的外部系统（哪些 API 可用、是否需要新接入）
- 数据敏感等级、合规要求
- 是否存在"AI 不可承担"的责任边界（如医疗诊断结论、法律意见）

**可行性判据**：

- 至少 3 条场景能在现有 capability + 1-2 个新插件内闭环
- 客户能提供至少 1 类高质量知识源
- 角色与审批边界清晰（不模糊"谁能做什么")

不通过 → 暂缓立项，避免做"看起来酷但落不了地"的业务包。

### 6.2 步骤 2：业务包架构设计（2-3 周）

输出物：

| 产物 | 内容 |
|------|------|
| 场景剧本表 | 每个剧本的 capability 序列、写操作草稿、引用要求 |
| 角色权限矩阵 | 谁能查什么、谁能起草、谁能审批 |
| 知识源清单 | 来源、责任人、更新频率、敏感等级 |
| 插件清单 | 要新建的插件、复用的插件 |
| 风险与红线 | 哪些动作必须人工签字、哪些必须拦截 |
| 评估集大纲 | 每类样本量、采集方式 |

参考[制造业业务包文档](制造业设备运维助手-业务包开发文档.md) §3-§5。

### 6.3 步骤 3：插件开发与契约定义（3-6 周，并行）

每个插件按"contract-first"开发：

```text
1. 写 capability schema (input / output / risk / scope)
2. mock 实现 + 合约测试
3. 接外部系统真实客户端
4. 集成测试 + 幂等键 / 错误翻译
5. 沙箱租户验收
```

平台提供 `agent-cli plugin init / validate / test / pack` 工具链。

### 6.4 步骤 4：业务包配置与提示词（2-3 周）

平台侧不写代码，全部为 yaml + 文本：

1. 写 `package.yaml`：元数据、入口意图、默认策略
2. 写 `capability-map.yaml`：capability + tool 白名单、意图→动作映射（含 skill 引用，详见 §5）
3. 写 `skills/*.yaml`：抽取出可复用的剧本片段
4. 写 `knowledge-bindings.yaml`：知识源 + 扩展属性 schema + 过滤模板
5. 写 `approval_policy.yaml`：阈值规则
6. 写 `prompts/`：行为准则、规划提示词、安全提示模板
7. 平台 `PackageLoader` 自动校验 schema、生成必要索引（详见 §8）

### 6.5 步骤 5：评估集构建与回归（2 周）

| 评估集 | 推荐样本量 | 来源 |
|------|------|------|
| 黄金问答集 | 50-100 | 真实历史咨询 |
| 写操作样本 | 30-80 | 真实业务单据 |
| 红线对抗集 | 40-100 | 行业专家构造 |

红线对抗集**必须 100% 通过**才允许进入灰度。

### 6.6 步骤 6：沙箱 → 灰度 → 全量（4-12 周）

```text
沙箱租户验证（2 周）
 → 单业务线 / 单门店 / 单网点灰度（2-4 周）
 → 多点灰度（4 周）
 → 全量发布
```

每个阶段配置自动回滚条件（点踩率、写操作失败率、红线漏过等）。

## 7. 运行原理与链路

### 7.1 整体调用链

```text
用户提问
 → API Gateway 鉴权 / 限流
 → Supervisor 创建 RequestContext (tenant_id, business_package, user_scopes)
 → PackageLoader 取出 business_package 的 RuntimeView
   ├ capability 白名单
   ├ 知识源绑定与过滤模板
   ├ 提示词
   ├ 默认 Planner 策略
   └ 审批策略
 → 用 user_scopes 二次过滤可用 capability
 → 输入安全检查
 → 意图分类（限定在 entry_intents 内）
 → Memory 召回 + 知识检索（按业务包 filter_template）
 → Planner 生成执行计划（capability 池 = 该意图的映射）
 → Governance 评估（风险、配额、审批）
 → Executor 调度插件 / 模型 / 检索
 → Response Composer 组装回答 + 引用 + 草稿动作
 → Output Guard（业务包红线规则）
 → 返回用户 + 异步审计 / 评估
```

每一步都可被 Trace 还原，前端/审计台可查询"为什么这次只能用这些能力、这条引用从哪来、为什么走了审批"。

### 7.2 业务包装配视图

`PackageLoader` 在两类时机工作：

| 时机 | 动作 |
|------|------|
| 平台启动 | 加载所有 `released` 业务包 → 缓存 PackageRuntimeView |
| 业务包发布/变更 | 比对 schema diff → 自动 ALTER（仅加列加索引）→ 热更新视图 |
| 单次请求 | 从缓存取视图 + 用 user_scopes 过滤 → 注入 Supervisor |

**关键不变量**：

- 业务包之间彼此隔离，A 包的配置变更不影响 B 包
- 业务包不能直接读写平台核心表，只能通过 capability + SDK
- 业务包不能注入自定义中间件，所有治理逻辑走平台 Runtime

### 7.3 数据隔离原理

```text
租户 (tenant)
 └ 业务包发布 (tenant_package_release)
    └ 业务包视图 (PackageRuntimeView)
       ├ 能用哪些 capability
       ├ 能查哪些知识源
       └ 默认走什么策略

每次请求强制带 tenant_id；
检索 SQL 强制带 tenant_id 过滤；
插件调用前 PDP 校验 tenant + scope;
跨租户访问直接返回 TENANT_ISOLATION_VIOLATION。
```

## 8. 扩展机制（核心表零侵入）

这是行业能不断接入而不污染平台的关键设计。详见制造业业务包文档 §7.2、§9.1。

### 8.1 三层扩展模型

```text
平台固定列（所有行业共用，永不为某行业改动）
  + attributes JSONB（业务包按 schema 自由扩展）
  + 热点字段自动提升（高频过滤字段升级为 generated column + BTree 索引）
```

### 8.2 业务包声明扩展属性

每个业务包在 `knowledge-bindings.yaml` 自带 schema：

```yaml
chunk_attributes_schema:
  <field_name>:
    type: string | number | boolean | date | array<string> | enum
    indexed: hot | warm | cold
    filter: eq | in | lte | gte | array_contains
    description: 用于审计和管理台展示
```

平台根据 `indexed` 等级自动决定：

| 等级 | 索引方式 | 适用 |
|------|------|------|
| hot | generated column + BTree | 高频过滤、范围查询 |
| warm | JSONB GIN | 中频过滤、布尔/数组 |
| cold | 不建索引 | 仅展示 |

### 8.3 业务包声明检索过滤

```yaml
retrieval:
  filter_template:
    <intent_name>:
      - field: <chunk_attribute>
        op: <eq/in/lte/...>
        from_context: <context_field> | value: <literal>
      - any_of:
          - ...
```

平台 `AttributeFilterCompiler` 翻译为安全 SQL，业务包不写 SQL。

### 8.4 新增业务对象表

如果行业必须有自己的"业务事件聚合表"（如制造业 `mfg_incident`、医疗 `med_episode`、金融 `fin_case`），按命名空间在业务包目录下提供 migration：

```text
packages/<industry>/migrations/
├── 0001_create_<industry>_<entity>.sql
└── ...
```

平台校验：

- 表名前缀必须是业务包命名空间
- 不允许 ALTER 平台核心表
- 不允许跨业务包外键

### 8.5 接入新行业的零侵入清单

| 接入项 | 改平台代码 / 表？ |
|------|------|
| 增加扩展属性字段 | 否，写 `chunk_attributes_schema` |
| 扩展属性建索引 | 否，按 `indexed` 自动生成 |
| 新增检索过滤规则 | 否，写 `filter_template` |
| 新增 capability | 否，注册插件包 |
| 新增剧本意图 | 否，写 `capability-map.yaml` |
| 新增审批阈值 | 否，写 `approval_policy.yaml` |
| 新增提示词 / 红线 | 否，写 `prompts/` + OutputGuard 规则 |
| 行业专属业务对象表 | 是，但限定在业务包命名空间，不动核心表 |

## 10. 治理、评估与上线

### 10.1 通用治理

所有业务包必须满足：

| 维度 | 要求 |
|------|------|
| 引用 | 所有事实性回答必须引用知识源；无来源则明示 |
| 写操作 | 默认草稿，关键动作必须人工确认或审批 |
| 审计 | 写操作 100% 留痕，不参与采样丢弃 |
| 数据 | 强制租户隔离；敏感字段按角色脱敏 |
| 红线 | 行业红线在 OutputGuard 强制，不能依赖提示词 |
| 模型 | 高敏感场景默认走私有模型；不允许跨境路由 |

### 10.2 通用评估

| 评估类型 | 通用指标 |
|------|------|
| 准确率 | 黄金集准确率 ≥ 90% |
| 引用率 | 事实性问题引用率 ≥ 90% |
| 红线 | 红线对抗集拦截率 100% |
| 时延 | 首 token P95 ≤ 3s，完整完成 P95 ≤ 10s |
| 成本 | 单次会话平均成本控制在租户预算阈值内 |

### 10.3 通用上线 SOP

```text
1. 沙箱租户回归通过 + 红线 100%
2. 客户业务方书面验收
3. 选择灰度对象（业务线 / 网点 / 门店 / 角色）
4. 配置自动回滚阈值
5. 第一阶段灰度 1-2 周 → 复盘 → 扩量
6. 全量后持续监控指标 4 周
7. 进入常态运营，按月做评估回归
```

## 12. 常见问题与设计取舍

### 12.1 为什么不做"行业大模型"，而做业务包？

行业知识更新快、规则版本多、合规边界刚性。把这些塞进模型权重，迭代成本极高，且不可解释。业务包把"规则、知识、流程"留在配置和插件里，模型只负责生成与组合，**易迭代、易审计、易回滚**。

### 12.2 业务包能不能调用其他业务包的 capability？

默认不能。每个业务包是封闭装配单元，跨包调用要通过通用包（如 `kb_qa`、`hr_general`）走。如确有跨包需求，在租户层显式启用多个业务包，由用户上下文切换。

### 12.3 插件能不能直接读平台核心表？

不能。所有数据访问通过 capability + SDK，平台通过 PDP 校验。插件直接读表会绕过审计、租户隔离、敏感字段脱敏，是平台禁止的。

### 12.4 多个业务包同时启用会冲突吗？

不会。业务包之间通过 `entry_intents` 路由区分；意图模糊时由 Planner 选择最匹配的业务包；用户也可在工作台显式切换。配置层面通过 §13.5 的"层级覆盖"解决：业务包级 → 租户级 → 请求级。

### 12.5 业务包升级会破坏旧数据吗？

不会。`chunk_attributes_schema` 升级只允许加字段或加索引，不允许 DROP / ALTER TYPE；ALTER 由平台产出，业务包开发者不写 DDL。回滚时把生成列设为不可见，不删数据。

### 12.6 如何评估某个行业值不值得做业务包？

三个判据：

1. 至少 5 条高频场景能在 1-2 个新插件 + 现有能力内闭环
2. 客户能持续提供高质量知识（不是一次性导入）
3. 规则边界清晰，不存在"AI 不能承担的责任真空"

满足 ≥ 2 条才立项。

### 12.7 Skill 与插件 Capability 怎么选？

判据是"组合是否稳定 + 是否需要独立审计单元"：

- 组合在多剧本里重复、内部逻辑稳定 → **抽 Skill**（业务包内 yaml 即可）
- 需要新接外部系统、有独立的写副作用、跨业务包共享 → **开新 Capability + 插件**

错误倾向：把"调几个 capability + 几行胶水"硬塞成一个新插件，会让插件越来越碎；反过来把"写操作"塞进 Skill 也错——Skill 没有独立的审批路径。

### 12.8 谁负责业务包的长期维护？

推荐"双负责人制"：

- 业务方负责人：剧本、规则、知识更新、评估集维护
- 技术负责人：插件、模型路由、性能、灰度回滚

每月一次联合复盘，季度做版本升级评审。

---

**结语**

本平台的核心信念：**平台沉淀治理，行业沉淀规则，模型负责理解与生成**。把这三件事分开后，新行业接入 = 写配置 + 写插件 + 训评估，工作量大头在"业务专家与领域知识"，而不是"工程改造"。这正是行业业务包能持续低成本扩展的原因。

下一步建议：

1. 在仓库新建 `packages/` 顶层目录与 `agent-cli package init` 脚手架
2. 选择一个业务方需求强烈的行业（推荐金融或政务）按本方案试做第二个业务包，验证通用流程
3. 把首两个业务包（制造业 + 第二个行业）的共性沉淀回本文档，形成 v1.1
