## 1. 方案目标

本文档用于说明在当前系统“知识库治理中心”页面基础上，如何承接跨行业、多格式、插件化索引与混合召回能力的前端交互。

目标包括：

- 不重做页面体系，基于当前 `Shell + page-section + stat-card + panel-card + stack-list` 风格扩展。
- 让知识入库从“粘贴 Markdown / 纯文本”升级为“文件 / 行业 / 插件 / 索引策略”可配置流程。
- 让用户能看到每份知识源进入了哪些索引：全文、向量、表格、实体、关系、API、错误码等。
- 让业务管理员能按行业插件配置索引 Profile，但不接触底层数据库。
- 让检索质量、召回路径、重建任务、权限过滤可观测。

## 3. 交互设计原则

1. 不让业务用户理解底层库。页面展示“索引类型”和“召回通道”，不暴露 PostgreSQL、pgvector、OpenSearch 等内部细节。
2. 行业能力来自插件。用户选择行业 / 业务域后，页面自动加载该行业插件的 `KnowledgeProfile`。
3. 入库流程分步展示。避免一个大表单堆满所有高级配置。
4. 默认值来自插件 Profile。业务用户只需要确认或微调，不必手填实体类型、权重和索引路由。
5. 所有高风险动作都有预检。比如全量重建、跨租户数据源、权限等级变更、废弃文档恢复。
6. 所有检索结果可解释。能看到命中了哪个索引、哪个 chunk、哪个实体、哪个插件 Profile。

## 5. 知识入库交互

### 5.1 当前入库表单升级

当前 `KnowledgeIngestPanel` 包含：

- 知识源名称
- 负责人
- 文档内容
- 创建索引按钮

建议升级为分步表单：

```text
Step 1 选择数据源
Step 2 选择行业与插件
Step 3 解析预览
Step 4 索引策略确认
Step 5 提交入库并观察流水线
```

### 5.2 Step 1：选择数据源

字段：

| 字段 | 控件 | 说明 |
|------|------|------|
| 知识源名称 | input | 必填 |
| 负责人 | input / user picker | 默认当前用户或知识平台组 |
| 来源类型 | segmented control | 文件上传、文本粘贴、URL、Confluence、SharePoint、API |
| 文件类型 | 自动识别 + select | PDF、Word、Excel、CSV、PPT、Markdown、HTML、JSON、图片 |
| 文档状态 | select | draft、published |
| 权限等级 | select | 公开、部门、租户、敏感 |

交互：

- 用户上传文件后，前端展示文件类型、大小、页数 / sheet 数 / slide 数预估。
- 如果是文本粘贴，保留当前 textarea 交互。
- 如果是 Excel / CSV，下一步展示 sheet 和表头预览。
- 如果是 PDF / Word，下一步展示章节识别预览。

### 5.3 Step 2：选择行业与插件

字段：

| 字段 | 控件 | 说明 |
|------|------|------|
| 行业 | select | finance、legal、hr、manufacturing、customer_service、government、general |
| 业务域 | select | 从行业插件 `business_domains` 加载 |
| 文档类型 | select | policy、manual、faq、api、rule_table、contract、regulation |
| 知识插件 | select / readonly | 根据行业自动匹配，如 `FinancePlugin` |
| Profile 版本 | select | 用于灰度和回滚 |

交互：

- 行业选择后，请求插件 Profile。
- 自动填充支持文件类型、实体类型、推荐索引通道。
- 如果文件类型不在插件支持范围内，显示“可继续使用通用 Profile”。
- 用户可以选择“通用知识 Profile”作为 fallback，但页面应提示行业实体抽取能力会降低。

### 5.4 Step 3：解析预览

按文件类型展示不同预览：

| 文件类型 | 预览内容 |
|----------|----------|
| PDF / Word | 标题层级、页码、段落样例、表格数量 |
| Excel / CSV | sheet 列表、表头、前 10 行、字段类型推断 |
| PPT | slide 缩略信息、标题、OCR 文本数量 |
| Markdown / HTML | heading tree、代码块、链接 |
| API / JSON | path / schema / key path |
| 图片 | OCR 文本、版面区域 |

用户操作：

- 勾选需要入库的章节、sheet、slide。
- 修改自动识别错误的表头或文档类型。
- 确认是否抽取表格行、实体和关系。

### 5.5 Step 4：索引策略确认

展示插件生成的索引路由：

```text
将写入：
  -> document_index
  -> chunk_index
  -> fulltext_index
  -> vector_index
  -> table_index
  -> entity_index
  -> relation_index
```

推荐 UI：

- 使用 checklist 展示索引通道。
- 默认选中插件 Profile 推荐项。
- 高级项折叠展示，如 Reranker 权重、chunk 大小、是否启用关系抽取。

不同角色：

- 普通业务管理员：只能使用默认策略。
- 平台管理员：可以调整 chunk 策略、索引通道、重排权重。
- 审计人员：只读查看。

### 5.6 Step 5：提交与流水线反馈

提交后不应只显示“入库成功”，而应显示任务状态。

任务阶段：

```text
上传完成
 -> 文件解析
 -> 行业插件增强
 -> chunk 生成
 -> 实体抽取
 -> 关系抽取
 -> 全文索引
 -> 向量索引
 -> 质量校验
 -> 发布
```

状态：

- 等待中
- 运行中
- 成功
- 部分成功
- 失败
- 已回滚

失败时展示：

- 失败阶段
- 原因
- 可重试动作
- 是否影响已发布索引

## 8. 召回调试交互

新增“召回调试”页签，给平台管理员和插件开发者使用。

### 8.1 Query 调试入口

字段：

| 字段 | 控件 | 说明 |
|------|------|------|
| Query | textarea | 用户问题 |
| 行业 | select | 可自动识别，也可手动指定 |
| 业务域 | select | 来自 Profile |
| 插件 Profile | select | 支持灰度版本对比 |
| top_k | stepper | 默认 10 |
| 是否启用 Reranker | toggle | 默认开启 |

按钮：

- 运行召回
- 对比 Profile
- 保存为评测样例

### 8.2 多路召回结果

结果按通道展示：

```text
全文召回
向量召回
表格召回
实体召回
关系召回
API / 错误码召回
融合后结果
Reranker 后结果
```

每条结果展示：

- 标题
- source type
- 命中索引
- score / rank
- chunk 摘要
- 来源定位：页码、sheet、row、slide、API path
- 权限过滤原因

### 8.3 召回解释

前端应展示一条可解释链路：

```text
用户 query
 -> 识别行业 finance
 -> 命中 FinancePlugin v1.0.0
 -> preferred_indexes: table_rows, entities, fulltext, vector
 -> filters: tenant_id, industry, business_domain, status
 -> RRF 融合
 -> Reranker 重排
```

这能帮助插件开发者判断问题出在：

- 行业识别
- Profile 配置
- 索引缺失
- 权限过滤
- Reranker 排序
- 文档本身缺失

## 12. 权限与安全交互

知识库治理页面涉及上传、重建、下线、Profile 变更，需要根据角色控制。

| 动作 | 普通业务管理员 | 平台管理员 | 审计人员 |
|------|----------------|------------|----------|
| 上传数据源 | 是 | 是 | 否 |
| 修改行业 Profile | 否 | 是 | 否 |
| 运行召回调试 | 只限本租户 | 是 | 只读 |
| 全量重建 | 否 | 是 | 否 |
| 下线数据源 | 需审批 | 是 | 否 |
| 查看审计日志 | 部分 | 是 | 是 |

高风险操作：

- 全量重建
- Profile 发布
- 下线数据源
- 修改权限等级
- 跨租户数据源绑定

这些动作应触发确认弹窗或审批草稿。

## 13. 前后端接口建议

### 13.1 当前接口

当前前端已使用：

```text
GET  /admin/knowledge
POST /admin/knowledge/ingest
```

当前 `POST /admin/knowledge/ingest` payload：

```json
{
  "name": "产品实施手册",
  "owner": "知识平台组",
  "source_type": "Markdown",
  "content": "..."
}
```

### 13.2 建议扩展接口

```text
GET  /admin/knowledge/profiles
GET  /admin/knowledge/profiles/{industry}
POST /admin/knowledge/parse-preview
POST /admin/knowledge/ingest
GET  /admin/knowledge/tasks/{task_id}
GET  /admin/knowledge/sources/{source_id}
POST /admin/knowledge/sources/{source_id}/reindex
POST /admin/knowledge/retrieval-debug
GET  /admin/knowledge/evaluations
POST /admin/knowledge/evaluations/run
```

### 13.3 入库 payload 建议

```json
{
  "name": "企业授信审批规则",
  "owner": "风控平台组",
  "source_type": "file",
  "file_type": "xlsx",
  "industry": "finance",

  "doc_type": "rule_table",
  "plugin_profile": "FinancePlugin@1.0.0",
  "status": "published",

    "extract_entities": true,
    "extract_relations": true,
    "build_fulltext_index": true,
    "build_vector_index": true

}
```

### 13.4 召回调试 payload 建议

```json
{
  "query": "小微企业授信额度上限是多少？",
  "industry": "finance",
  "business_domain": "credit_approval",
  "plugin_profile": "FinancePlugin@1.0.0",
  "top_k": 10,
  "rerank": true,
  "debug": true
}
```

返回结构建议：

```json
{
  "query_analysis": {
    "industry": "finance",
    "entities": ["小微企业"],
    "preferred_indexes": ["table_rows", "entities", "fulltext", "vector"]
  },
  "results_by_index": {
    "table_rows": [],
    "entities": [],
    "fulltext": [],
```

## 15. 总结

前端交互的核心不是把所有索引细节暴露给用户，而是把复杂能力包装成三个可理解动作：

1. 入库时选择“行业、业务域、插件 Profile、索引策略”。
2. 治理时查看“数据源、流水线、索引产物、质量状态”。
3. 调试时解释“query 如何经过行业插件、哪些索引被召回、为什么最终排在前面”。

这样既能延续当前知识库治理页面的结构，又能承接跨行业、多格式、插件化索引和混合召回的后续演进。
