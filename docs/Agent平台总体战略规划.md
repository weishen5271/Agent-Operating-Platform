# Agent 业务操作平台 — 总体战略规划

> 产品演进 · 技术架构 · 商业化路径

---

## 一、战略定位

### 核心定位

> 外挂式 Agent 业务操作平台 —— 连接企业已有业务系统、知识库和工具能力，在权限、租户、风险和审计约束下，围绕业务对象执行 AI 分析、建议生成、草稿创建和成果沉淀。

平台的价值在于：让企业已有的业务系统获得 AI 增强能力。企业不需要替换现有系统，不需要迁移数据，只需要接入平台就能让已有业务流程拥有 AI 分析、决策辅助和自动化执行的能力。

### 平台核心能力

| 能力 | 说明 |
|---|---|
| 业务包驱动 | 读取业务包声明的业务对象、AI 动作、Skill、Capability、Tool，动态加载能力 |
| 外部系统按需调用 | 通过 API / MCP / HTTP executor 按需查询外部业务系统，获取实时业务数据 |
| 知识增强分析 | 结合 RAG / Wiki / 知识库，为 AI 分析提供领域知识支撑 |
| 过程与成果沉淀 | 保存 AI 执行过程（Trace）、业务成果（BusinessOutput）、审批草稿（DraftAction） |
| 统一治理 | 提供权限、租户、风险控制、输出审查（OutputGuard）和全链路审计能力 |

### 交付形态双轨策略

平台同时支持两种交付形态，共享同一个 AI 执行引擎：

- **独立工作台**：平台自有前端，用户直接登录使用，适合演示、轻量场景和早期客户验证。
- **嵌入式模块**：作为客户已有业务系统的 AI 插件/子模块，数据访问变为系统内部调用，落地门槛更低。

两种形态的核心区别不在于能力，而在于用户触达方式和集成深度。引擎层完全一致，交付层按场景选择。

## 二、两种交付形态详细设计

### 形态一：独立工作台

#### 定位与适用场景

独立工作台是平台的"自营门面"，用户通过平台自有的 Web 界面直接使用 AI 能力。适用于以下场景：

- **早期验证阶段**：客户还没决定深度集成，先用独立工作台体验和评估平台价值。
- **轻量使用场景**：客户只有少量用户需要 AI 辅助，不值得做系统集成。
- **跨系统分析场景**：用户需要同时调用多个外部系统的数据做综合分析，独立工作台提供统一操作界面。
- **内部运营场景**：平台运营团队用工作台配置、测试和调试业务包。

#### 用户路径

```
登录平台
  → 选择业务包
  → 选择业务对象类型
  → 输入或检索业务对象 ID
  → 选择 AI 动作
  → 平台按需调用外部系统和知识库
  → 生成分析、建议、报告或草稿
  → 保存为 BusinessOutput / DraftAction
  → Trace 审计全过程
```

#### 页面结构

| 页面 | 功能 |
|---|---|
| AI 工作台 | 核心操作页面，选择业务包、对象、动作，执行并查看结果 |
| 业务包详情 | 展示业务对象列表、AI 动作列表、依赖能力状态、配置完整性 |
| 业务成果 | 展示 AI 生成的成果，关联来源 run、对象、Trace、草稿 |
| Chat | 自然语言探索入口，适合不确定意图的交互，可触发结构化 AI Action |
| 审批中心 | 展示待确认的 DraftAction，支持审批确认或驳回 |

#### 技术要点

- 前端组件（对象输入、动作选择、结果展示、Trace 查看）设计为独立组件，不绑定页面布局。
- 认证方式支持用户名密码登录和 SSO。
- Chat 页保留但降级为入口之一，AI 工作台是更明确的业务操作入口。

### 形态二：嵌入式模块

#### 定位与适用场景

嵌入式模块是把平台的 AI 能力"塞进"客户已有的业务系统里，用户在自己熟悉的系统中直接使用，不需要切换工作环境。适用于以下场景：

- **深度集成客户**：客户希望 AI 能力成为自己系统的一部分，而不是一个外部工具。
- **数据敏感客户**：客户不愿意把业务数据通过外部 API 暴露，嵌入后数据访问变为内部调用。
- **高频使用场景**：用户每天大量使用 AI 辅助，需要和业务操作无缝衔接。
- **OEM/白标场景**：合作伙伴把平台能力包装到自己的产品中。

#### 集成模式

平台提供三种标准集成模式，客户按技术能力和需求选择：

**模式一：iframe 嵌入（最轻量）**

客户系统通过 iframe 加载平台的工作台组件页面。平台提供可配置的嵌入页面 URL，支持通过参数预设业务包、对象类型等上下文。

```
适用场景：客户系统是 Web 应用，有 iframe 容器位置
集成成本：低，前端加一个 iframe 即可
能力范围：完整的工作台交互体验
限制：样式隔离依赖 iframe 沙箱，跨域通信需要 postMessage
```

技术实现要点：
- 平台提供 `/embed/workbench` 嵌入专用页面，去掉平台导航和登录框。
- 支持 URL 参数预设上下文：`?package_id=xxx&object_type=xxx&object_id=xxx`。
- 认证通过 token 参数或 postMessage 传递，不要求用户二次登录。
- 支持 postMessage 双向通信：宿主系统可以向嵌入页推送上下文变更，嵌入页可以向宿主系统推送执行结果。

**模式二：Web Component 嵌入（中等集成度）**

平台核心组件打包为标准 Web Component，客户系统直接在自己的页面中使用自定义标签。

```
适用场景：客户系统需要更深度的 UI 集成，iframe 不够灵活
集成成本：中等，需要引入组件库
能力范围：可按需使用单个组件（如只嵌入动作执行面板）
限制：需要处理样式隔离（Shadow DOM），组件版本管理
```

技术实现要点：
- 核心组件（对象输入、动作选择、结果展示、Trace 查看）各自打包为独立 Web Component。
- 使用 Shadow DOM 实现样式隔离，嵌入到客户系统不产生样式冲突。
- 组件通过属性（attributes）接收配置，通过自定义事件（CustomEvent）向外推送状态变更。
- 提供 CDN 托管的组件包和 npm 包两种分发方式。

示例用法：
```html
<agent-workbench
  api-endpoint="https://api.agent-platform.com"
  api-key="ak_xxx"
  package-id="industry.mfg_maintenance"
  object-type="equipment"
  object-id="CNC-01"
  @action-completed="handleResult"
/>
```

**模式三：后端 SDK / API 直接调用（最灵活）**

客户系统的后端直接调用平台 API，自己控制 UI 展示。平台只提供执行引擎能力，不参与前端渲染。

```
适用场景：客户有自己的前端团队，只需要 AI 执行能力
集成成本：高，客户需要自行开发 UI
能力范围：最灵活，完全自定义交互体验
限制：客户承担 UI 开发成本
```

技术实现要点：
- 平台提供完整的 REST API，覆盖动作查询、执行、结果获取、Trace 查询全链路。
- 提供 Python / Java / TypeScript SDK 封装常用调用逻辑。
- API 认证使用 API Key + 租户隔离，支持 OAuth 2.0 授权码流程。
- 接口返回自包含的结构化数据，客户系统拿到就能渲染，不依赖平台任何前端资源。
- 提供 Webhook 回调，支持异步执行场景（长时间运行的 AI 动作）。

#### 嵌入式认证方案

| 认证方式 | 适用集成模式 | 说明 |
|---|---|---|
| Token 传递 | iframe | 宿主系统生成临时 token，通过 URL 参数或 postMessage 传给嵌入页 |
| API Key | Web Component / SDK | 租户级 API Key，在组件属性或 SDK 初始化时配置 |
| OAuth 2.0 | SDK / API | 标准授权码流程，适合需要用户级权限的场景 |
| SSO 集成 | 所有模式 | 与客户的企业 SSO 对接，实现免登录 |

#### 嵌入式数据流

嵌入模式下，数据流和独立工作台有一个关键区别：外部系统的数据可以由宿主系统直接提供，而不需要平台去调用外部 API。

```
独立工作台模式：
  平台 → 调用外部系统 API → 获取数据 → AI 分析

嵌入式模式（宿主推送）：
  宿主系统 → 将业务数据推送给平台 → AI 分析

嵌入式模式（平台拉取）：
  平台 → 调用宿主系统内部 API → 获取数据 → AI 分析
```

这意味着嵌入模式可以绕过"外部系统 API 难开放"的问题：客户系统自己把数据喂给平台，平台只负责 AI 分析和结果输出。

#### 两种形态的能力对比

| 维度 | 独立工作台 | 嵌入式模块 |
|---|---|---|
| 用户体验 | 独立界面，需要切换系统 | 嵌入在业务系统中，无缝使用 |
| 集成成本 | 零集成，开箱即用 | 需要客户 IT 团队配合 |
| 数据获取 | 平台主动调用外部 API | 宿主系统推送或平台内部调用 |
| 适用客户 | 早期客户、轻量场景、演示验证 | 深度客户、高频场景、OEM |
| 品牌呈现 | 平台品牌 | 可白标，呈现为客户系统的功能 |
| 部署方式 | SaaS 或私有化 | 跟随客户系统部署 |
| 演进优先级 | Phase 1 立即做 | Phase 3 就绪，有客户需求时落地 |

## 三、产品演进路线

### Phase 1：引擎抽取 + 最小闭环验证（0–3 个月）

> 目标：把可复用的 AI 执行内核从 ChatService 中抽出来，用一个 stub 场景端到端跑通，验证架构合理性。

**产品交付物：**

- AI 工作台最小版（选业务包 → 输入对象 ID → 选动作 → 查看结果）
- "设备故障分析"场景完整跑通（全 stub）
- BusinessOutput 和 Trace 可查看
- Chat 和工作台共享同一执行引擎

**为嵌入形态预留的工作：**

- 前端核心组件设计为独立组件，不绑定页面布局和全局路由
- API 同时支持会话登录态和 API Key 认证
- 接口返回结构自包含

**里程碑：**

- 执行内核不依赖 ChatService 任何代码即可独立运行
- API Key 认证方式能正常调用全部接口

---

### Phase 2：声明完善 + 审批闭环（3–6 个月）

> 目标：业务包自描述能力完整，高风险动作进入审批流程，产品从"能跑通"变为"能用起来"。

**产品交付物：**

- 业务包详情页展示业务对象、AI 动作、依赖能力状态
- 外部对象检索（lookup_capability）
- 高风险动作生成 DraftAction 进审批
- 业务成果页展示来源 run、对象、Trace、关联草稿

**为嵌入形态预留的工作：**

- 嵌入专用页面 `/embed/workbench` 开发，支持 URL 参数预设上下文
- postMessage 通信协议定义
- API 文档和调用示例

**里程碑：**

- 至少一个业务包的 manifest 通过严格 JSON Schema 校验
- 一个客户场景的外部系统真实接通（替换 stub）

---

### Phase 3：Chat 融合 + 嵌入就绪（6–12 个月）

> 目标：Chat 和 AI Run 融合，嵌入式交付能力就绪，在真实客户场景中验证价值。

**产品交付物：**

- Chat Planner 识别意图并路由到结构化 AI Action
- 核心组件打包为 Web Component（Shadow DOM 样式隔离）
- iframe 嵌入方案完整可用
- 后端 SDK（至少 Python + TypeScript）
- 嵌入集成文档和示例项目
- OAuth 2.0 授权支持

**里程碑：**

- 第一个付费客户在真实工作流中每天使用
- 至少一个客户以嵌入模式完成集成

---

### Phase 4：生态扩展 + 规模化（12–24 个月）

> 目标：从单一场景扩展到多行业多场景，形成业务包生态。

**产品交付物：**

- 业务包开发规范和模板库
- 业务包市场（内部或合作伙伴）
- 多行业覆盖（制造业、能源、物流、医疗等）
- Agent 编排能力（多步骤、多动作串联）
- 客户自助配置业务包和 AI 动作
- 白标能力（嵌入模块去除平台品牌标识）
- Webhook 回调支持异步长时间执行

**里程碑：**

- 3 个以上行业场景落地
- 合作伙伴能独立开发业务包

## 四、技术架构演进

### 架构分层

整体架构分为三层，引擎层是核心，交付层可替换：

| 层次 | 包含内容 | 设计原则 |
|---|---|---|
| 交付层 | AI 工作台前端、Chat UI、嵌入页面、Web Component、SDK、API Gateway | 可替换，不包含业务逻辑 |
| 引擎层 | AI Execution Engine、业务包运行时、Skill/Capability 编排、LLM 调用、Trace、OutputGuard | 核心资产，所有交付形态共享 |
| 基础层 | PostgreSQL、租户/权限、插件配置、知识库 | 稳定不动，服务于上层 |

### 关键技术决策

**1. 执行内核抽取**

从 ChatService 中抽取与"聊天会话"无关的执行逻辑，形成 AI Execution Engine。判断标准：如果将来要做嵌入式 SDK，不需要带上任何 Chat 相关的代码就能跑通一次 AI 动作执行，说明抽取到位了。

**2. 接口设计双模式**

所有 API 同时支持会话登录态和 API Key 认证。接口返回自包含，客户系统拿到结果就能渲染，不依赖平台前端状态。

**3. Manifest 契约化**

业务包 manifest 是平台核心契约。尽早定义 JSON Schema 并做严格校验。manifest 只描述业务语义，不包含 UI 渲染指令。同一个业务包在工作台里是一种呈现，嵌入到客户系统里可以是另一种呈现。

**4. Stub 体系标准化**

定义统一的 stub executor 接口和返回格式。所有 stub 返回必须带 `source: stub` 标记。Trace 中记录每个调用是 real 还是 stub。前端对 stub 结果有显眼标识。

**5. 前端组件可独立打包**

对象输入、动作选择、结果展示、Trace 查看等核心组件不依赖整体页面布局和全局路由状态。组件通过属性接收配置、通过事件向外推送状态。将来可单独导出为 Web Component 或 iframe 页面。

**6. 数据输入通道抽象**

AI 执行引擎获取外部数据的方式不应硬编码为"平台主动调用外部 API"。设计一个数据输入通道抽象，支持三种模式：平台主动拉取（HTTP/MCP executor）、宿主系统推送（嵌入场景下通过 API 传入）、混合模式（部分拉取部分推送）。这样同一个业务包在不同交付形态下都能正常工作。

### 技术演进时间线

| 时间 | 引擎层 | 交付层 | 基础层 |
|---|---|---|---|
| 0–3 月 | 抽取执行内核，新增 ai_run 表，数据输入通道抽象 | 工作台最小版，组件独立化 | 无变动 |
| 3–6 月 | Manifest 校验，审批流程 | 嵌入专用页面，postMessage 协议 | 可选分析快照表 |
| 6–12 月 | Chat 融合，Agent 编排 | Web Component 打包，SDK，OAuth | OAuth / SSO 支持 |
| 12–24 月 | 多步骤编排引擎 | 业务包市场，白标，自助配置 | 多区域部署 |

### 核心数据模型

- ai_run — AI 动作执行记录
- trace — 执行过程追踪
- business_outputs — AI 生成的业务成果
- draft_actions — 待审批的草稿动作
- conversations — Chat 对话记录
- 知识库 / Wiki
- 插件配置
- 必要的分析摘要快照

## 五、商业化路径

### 目标客户画像

最佳目标客户是"有系统但缺 AI"的企业：已经部署了 CMMS、ERP、MES 等业务系统，有一定数据基础，但缺少 AI 增强能力。这类客户的特征是：

- 有明确的业务痛点（故障分析慢、依赖老师傅经验、多系统信息散落）
- 有 IT 基础能力配合对接
- 愿意为 AI 能力付费，但不愿意替换现有系统

### 定价策略建议

| 模式 | 适用场景 | 定价逻辑 | 注意事项 |
|---|---|---|---|
| 订阅制 | 独立工作台形态 | 按租户 + 用户数 + AI 调用量 | 早期可用固定包月降低门槛 |
| 嵌入许可 | 嵌入式模块形态 | 按 API 调用次数或包月 | 客户看到的是自己系统的功能 |
| 项目制 | 早期大客户 | 定制开发 + 年费 | 用于积累行业 know-how |
| OEM/白标 | 合作伙伴分发 | 底价授权 + 分成 | 合作伙伴负责行业拓展 |

### 商业化分阶段

**第一阶段：免费验证（0–6 个月）**

目标不是赚钱，是找到第一个愿意深度配合的客户。免费提供平台能力，换取真实场景和反馈。这个阶段的核心产出是：一个端到端跑通的真实场景，和客户愿意继续用的口头确认。优先通过独立工作台交付，集成成本最低。

**第二阶段：标杆客户收费（6–12 个月）**

基于第一阶段的场景经验，包装成可复制的行业解决方案。开始收费，验证客户付费意愿和定价模型。同步推进嵌入式交付能力，为深度集成客户提供选项。目标：3–5 个付费客户，月经常性收入覆盖基本运营成本。

**第三阶段：规模化增长（12–24 个月）**

通过业务包生态和合作伙伴体系扩大覆盖范围。嵌入式交付和白标能力降低落地门槛，让合作伙伴能把平台能力包装到自己的产品中分发。目标：多行业覆盖，年收入达到一定规模。

### 市场定位与竞争策略

平台的竞争优势在于"让客户现有的系统拥有 AI 能力，且全过程可审计"。具体来说：

- **与通用 AI 助手（ChatGPT、Copilot）的区别**：平台围绕业务对象执行结构化动作，有审批、审计、权限控制，输出可沉淀为业务成果。
- **与数据中台/BI 工具的区别**：平台做的是基于外部数据的 AI 分析和决策辅助，不是数据存储和报表展示。
- **与行业垂直 SaaS 的区别**：平台是横向能力层，通过业务包适配不同行业，不重新做一套行业软件。
- **嵌入式交付是核心差异化**：竞品大多只提供独立产品形态，平台能嵌入客户已有系统，降低使用门槛和替换成本。

## 六、风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 外部系统 API 难开放 | 闭环跑不通，客户体验差 | 嵌入式交付让数据获取变为内部调用；优先找有 API 能力的客户；stub 保证演示能力 |
| 客户不信任 AI 建议 | 有能力但无人用 | Trace 透明化推理过程；DraftAction 让人确认后再执行；区分事实、推断、建议 |
| 业务包生态冷启动 | 只有平台没有内容 | 团队自己做前 3–5 个标杆业务包；总结规范后再开放合作伙伴 |
| 技术架构变形 | 多种交付形态导致代码分叉 | 第一阶段就抽取公共内核；交付层只做薄封装，不允许在交付层写业务逻辑 |
| LLM 质量不稳定 | 生成的分析和建议不可靠 | OutputGuard 审查；Trace 记录完整推理过程；展示置信度和数据来源 |
| 定价模型验证不足 | 免费用户多但无人付费 | 免费阶段设置明确时间上限；尽早验证付费意愿 |
| 嵌入集成成本高 | 客户评估后放弃集成 | 提供三种集成模式由简到繁；iframe 方案做到开箱即用 |

## 七、贯穿始终的核心原则

1. **引擎和交付解耦。** 任何新功能先问"这是引擎层的还是交付层的"，不要混在一起。
2. **Manifest 是核心契约。** 业务包的 manifest 决定了整个平台的能力表达，要严格校验，只描述业务语义。
3. **Stub 标注必须显眼。** 任何使用了占位数据的地方，用户和开发者都能一眼看出来。
4. **先跑通一个场景。** 一个场景端到端跑通的价值远大于十个半成品。
5. **用客户场景验证方向。** 架构再完美，没有客户愿意每天用就不算验证通过。
6. **嵌入优先于重建。** 帮客户在已有系统上加能力，而不是让客户来用一个新系统。
