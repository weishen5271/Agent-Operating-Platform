## 1. 文档概述

### 1.1 产品名称

可扩展 Agent 平台

英文系统名称：`Agent Operating Platform`

### 1.2 产品定位

面向企业内部智能化场景的通用 Agent 平台底座，提供统一的对话入口、任务编排、知识检索、记忆管理、插件治理、权限审计与运行时控制能力。

平台本身不绑定具体行业，行业能力以“业务包 / 插件包”的形式接入。  
例如：

- HR 业务包
- 法务业务包
- 财务业务包
- 客服业务包
- 工业业务包

### 1.3 产品边界

平台负责：

- 通用智能能力底座
- 插件协议与运行时治理
- 权限、审批、审计、评估体系
- 知识接入与业务系统接入框架
- 多租户与环境配置管理

平台不直接负责：

- 某个单一行业的完整业务逻辑实现
- 具体业务系统的主数据治理
- 安全关键、不可逆、高实时控制动作的自主执行

## 2. 背景与目标

### 2.1 业务背景

企业在引入 Agent 时，常见问题包括：

- 每个业务部门各自建设智能助手，能力重复
- 对话、检索、流程、权限、审计等底层能力无法复用
- 不同行业场景差异大，但底座能力相似
- 随着插件和系统接入增加，治理、评估与安全成本急剧上升
- 缺少统一的能力契约，导致规划结果不稳定、集成成本高

### 2.2 产品目标

建设一个统一的可扩展 Agent 平台，使企业能够在不改动核心框架的前提下，快速接入新的业务插件、知识源和系统能力，形成“通用底座 + 行业包”的可持续模式。

### 2.3 核心目标

1. 提供统一自然语言交互入口与任务编排能力
2. 沉淀通用能力，避免各业务线重复建设
3. 通过插件协议和能力契约支持业务快速扩展
4. 满足企业级权限、审计、安全、评估和多租户要求
5. 支持从问答型场景逐步扩展到受控执行型场景

### 2.4 阶段目标

#### P0：通用底座可用

- 上线统一对话入口
- 上线知识检索、记忆、规划、插件调用能力
- 上线权限、审计、监控基础能力

#### P1：行业包可扩展

- 支持多个业务包并行接入
- 上线插件市场/企业内部插件仓库
- 上线评估、回归和灰度发布能力

#### P2：受控执行与平台化运营

- 支持审批后执行
- 支持多租户运营与环境隔离
- 支持插件生态和平台级治理看板

### 2.5 Agent 自主度递进模型

除阶段目标外，平台在能力演进上遵循"自主度递进"的纵深轴。该轴独立于 P0–P2 阶段目标，描述某项业务能力在真实业务场景中可以被赋予多大的自主权。

| 级别 | 名称 | 系统行为 | 用户参与度 | 典型场景 |
|------|------|---------|-----------|---------|
| L1 | 建议型 | 仅输出结论、分析与建议，不调用任何外部写接口 | 用户完全决策 | 知识问答、归因分析、方案建议 |
| L2 | 草稿型 | 生成结构化草稿，由用户校对后提交至业务系统 | 用户必须显式确认 | 报销草稿、工单草稿、邮件草稿 |
| L3 | 审批型 | 按审批链走完流程后，由平台代为调用外部系统 | 用户发起 + 审批人决策 | 采购、开户、权限申请、费用调整 |
| L4 | 受控自动化 | 在明确边界和熔断条件内自主执行 | 用户仅在异常时介入 | 例行巡检、数据回灌、批量通知 |

规则：

- 平台通过风险分级（见 10.5）决定某能力在某租户下允许的最高自主度
- 新业务包接入默认从 L1 / L2 起步，经过一段时间评估后才可申请升级
- 任何 L3 以上能力必须具备审计、熔断、回滚三项基础保障

## 3. 目标用户

### 3.1 主要用户角色

| 角色 | 说明 | 核心诉求 |
|------|------|------|
| 业务使用者 | 具体业务部门员工 | 通过自然语言完成查询、分析、流程辅助 |
| 业务管理员 | 各行业包/业务包负责人 | 配置知识源、插件、业务规则、提示词 |
| 平台管理员 | 平台运维与治理角色 | 管理租户、权限、审计、插件与模型配置 |
| 集成开发者 | 接入系统和开发插件的工程师 | 有稳定协议、清晰扩展点、低耦合接入 |
| 安全与审计人员 | 风控、内控、合规角色 | 有权限边界、审批链路和可追溯记录 |

### 3.2 用户分类

#### 终端用户

- 关注答案是否可用
- 关注是否能减少重复操作
- 不关心底层模型和插件细节

#### 平台管理者

- 关注平台稳定性、安全性、成本和扩展效率
- 关注插件是否可控、可回滚、可评估

#### 插件开发者

- 关注能力协议、开发模板、调试方式、测试标准
- 希望新增业务能力时不需要修改核心框架

## 4. 产品愿景与价值

### 4.1 产品愿景

成为企业内部智能化建设的统一 Agent 底座，以标准化能力契约和插件机制承载多行业、多部门、多场景的智能需求。

### 4.2 核心价值

- 统一入口：减少多个助手并存带来的割裂体验
- 能力复用：记忆、检索、规划、审计等底层能力统一沉淀
- 快速扩展：新增业务包主要通过插件接入完成
- 治理可控：权限、审批、审计、评估、回滚统一纳管
- 平台演进：支持从问答到流程协同再到受控执行逐步演进

## 5. 典型使用场景

### 5.1 通用知识问答

用户输入：

`公司的差旅报销规则是什么？`

系统需要：

1. 识别问题类型
2. 检索相关知识文档
3. 返回结构化答案与来源
4. 支持继续追问

### 5.2 业务查询

用户输入：

`张三还有几天年假？`

系统需要：

1. 识别实体和业务意图
2. 调用对应业务包插件
3. 返回结果并标注数据来源

### 5.3 复杂分析

用户输入：

`为什么这个月华东区退货率上升？`

系统需要：

1. 识别分析维度与时间范围
2. 调用多个数据插件
3. 进行聚合、总结与解释
4. 输出可能原因与建议动作

### 5.4 受控流程协同

用户输入：

`帮我生成这次采购审批的草稿`

系统需要：

1. 校验权限
2. 生成草稿
3. 展示待确认内容
4. 审批后执行或提交外部系统

### 5.5 行业包扩展

业务方希望新增一个行业包，例如工业、法务或客服，系统需要：

1. 注册业务包
2. 配置插件与能力契约
3. 接入知识源与业务系统
4. 配置权限与评估规则
5. 通过灰度方式上线

## 7. 功能设计

### 7.1 功能总览

| 模块 | 功能 | 优先级 |
|------|------|------|
| 交互层 | 对话、多轮上下文、结果呈现 | P0 |
| 编排层 | 意图识别、任务规划、结果整合 | P0 |
| 记忆层 | 会话记忆、长期记忆、记忆治理、被遗忘权 | P0 |
| 检索层 | 文档检索、结构化检索、混合检索、知识入库管道 | P0 |
| 插件层 | 插件注册、调用、健康检查、依赖解析 | P0 |
| 契约层 | capability schema、参数校验、输出校验、MCP 映射 | P0 |
| 治理层 | 权限、审批、审计、风险分级、配额限流 | P0 |
| 安全层 | Prompt 防注入、DLP、输出脱敏、内容审查、合规 | P0 |
| 模型层 | 模型抽象、按任务路由、主备降级、版本灰度 | P0 |
| 运营层 | 离线评估、线上评估、错误归因、成本分析、反馈回流 | P1 |
| 生态层 | 企业内部插件仓库、行业包接入、开发者工具链 | P1 |

### 7.2 统一对话入口

#### 功能描述

提供统一的自然语言入口，让用户通过一个工作台访问不同业务包能力。

#### 功能点

- 多轮对话
- 上下文延续
- 会话历史
- 结果卡片化展示
- 引用和依据展示

### 7.3 任务编排模块

#### 功能描述

根据用户意图和上下文生成任务计划，调度通用能力与行业插件完成请求。

#### 功能点

- 意图识别
- 任务分解
- 任务依赖分析
- 任务并串行编排
- 失败回退和重试控制

#### 规划器策略

平台支持多种规划策略，可按业务包 / 场景配置：

| 策略 | 适用场景 | 说明 |
|------|---------|------|
| 单步直答 | 简单问答、FAQ | 不做分解，直接调用检索或模型 |
| Plan-and-Execute | 结构清晰的多步任务 | 先出完整计划再串行执行 |
| ReAct（边想边做） | 需要基于中间结果决定下一步 | 思考与工具调用交替，适合探索性任务 |
| Reflexion（反思） | 高准确率要求、写操作前 | 执行后自检，必要时重规划 |

#### 重规划与思考预算

- 支持定义"重规划触发条件"：工具失败 N 次、中间结果置信度低于阈值、用户中途追加约束
- 每个规划会话应设置 `max_steps` 和 `thinking_token_budget`，超限后强制收敛
- 规划器应显式输出"已知 / 未知 / 下一步"的中间状态，便于调试与审计

### 7.4 检索与知识增强模块

#### 功能描述

统一支持对知识库、文档库、FAQ、制度手册和结构化数据进行检索。

#### 功能点

- 全文检索
- 向量检索
- 混合检索
- 重排与过滤
- 引用返回

#### 知识入库管道

仅有"检索"不够，必须配套知识入库与治理管道：

1. **采集**：支持定时拉取、事件触发、手动上传三种方式
2. **解析**：按文档类型（Word / PDF / HTML / 表格 / Wiki）做结构化解析
3. **切分**：支持按语义段落、按标题层级、按固定窗口三种策略
4. **嵌入**：记录使用的 embedding 模型与版本，便于换模型后的批量重建
5. **标签化**：自动抽取租户、密级、业务域、生效时间、失效时间等元数据
6. **上架 / 版本**：每份知识具备版本号、发布人、上架时间
7. **失效 / 淘汰**：支持 TTL、条件失效（如制度被新文件覆盖）、手动下架
8. **质量评分**：根据命中率、用户反馈、人工标注维持知识质量分

#### 权限与新鲜度

- 检索必须带租户与数据密级过滤，禁止跨租户召回
- 返回的引用必须包含「来源、版本、更新时间」三要素
- 对时效性敏感的内容（价格、政策、人员信息）应标注"仅供参考，请以系统为准"

### 7.5 插件与业务包管理模块

#### 功能描述

将行业能力通过业务包/插件包方式接入平台，实现底座与业务解耦。

#### 功能点

- 插件注册
- 插件启停
- 插件版本管理
- 依赖管理
- 健康检查
- 业务包分组展示

### 7.6 能力契约管理模块

#### 功能描述

每个插件能力都必须以标准 schema 进行声明，便于 Planner 和运行时治理。

#### 功能点

- capability 定义
- 输入/输出 schema 管理
- 幂等性、副作用级别声明
- 权限要求声明
- 版本兼容管理

#### Capability Schema 必备字段

| 字段 | 说明 |
|------|------|
| name / version | 能力唯一标识与语义化版本 |
| description | 供 Planner 做能力选择的自然语言描述 |
| input_schema | JSON Schema，描述入参结构与约束 |
| output_schema | JSON Schema，描述返回结构，Planner 可据此串联 |
| side_effect | none / read / write / irreversible 四级 |
| idempotent | 是否幂等，影响重试策略 |
| risk_level | low / medium / high / critical（对应 10.5） |
| required_scopes | 调用该能力需要的权限 scope 列表 |
| cost_hint | 预估成本区间，用于成本敏感路由 |
| timeout_ms | 预估执行时长上限 |
| tenant_scope | 全平台 / 租户独占 / 业务包独占 |

#### 与开放协议的兼容

- 内部契约保持独立演进，但应与 **MCP（Model Context Protocol）** 保持**双向映射能力**
- 平台应支持导入外部 MCP Server 作为插件（只读或读写，取决于副作用级别）
- 平台自有插件应能导出为 MCP 规范，便于未来对接 IDE / 外部 Agent 生态
- 版本兼容策略：小版本向后兼容，大版本升级须经过"能力等价性测试"方可替换

### 7.7 运行时治理模块

#### 功能描述

对任务执行过程中的超时、重试、审批、取消、补偿进行控制。

#### 功能点

- 风险分级
- 任务超时控制
- 重试策略
- 人工审批点
- 失败补偿

#### 配额与限流

为避免单租户 / 单场景拖垮平台，所有执行链路都应受配额保护：

| 配额维度 | 说明 | 触发后行为 |
|---------|------|-----------|
| 租户 QPS | 每租户每秒请求上限 | 排队或降级返回"系统繁忙" |
| 租户日 Token 配额 | 按业务包分摊 | 达 80% 告警，100% 拒绝并提示管理员 |
| 租户月成本预算 | 含模型、检索、插件外部调用 | 达阈值触发告警、达上限走降级模型 |
| 单请求 Token 上限 | 防止异常长会话 | 截断并提示用户 |
| 单请求步数上限 | 防止规划器死循环 | 强制收敛，返回当前最优结果 |
| 插件调用频次 | 按 capability 维度 | 超限走熔断，N 分钟内不再调度 |

#### 熔断与降级

- 插件连续失败率超阈值自动熔断，熔断期间 Planner 跳过该能力
- 模型不可用时按 7.12 配置的主备链路降级
- 检索服务不可用时回退为"仅模型回答 + 明确声明未命中知识库"
- 所有降级行为必须在回答中对用户可见或在审计中明确标注

### 7.8 权限与审计模块

#### 功能描述

保障平台在多租户、多角色、多业务包环境下安全运行。

#### 功能点

- RBAC / ABAC 权限控制
- 数据范围控制
- 审计日志
- 敏感操作留痕
- 按租户查看操作记录

### 7.9 评估与运营模块

#### 功能描述

对平台质量、成本、稳定性和插件效果进行持续评估。

#### 功能点

- 评估集管理
- 回归执行
- 指标看板
- 成本监控
- 插件效果分析

#### 离线评估

- 每个业务包维护自己的评估集（标准问答集 + 标注标签）
- 支持基于 LLM-as-Judge 的自动化评分（准确性、引用正确性、回答完整性）
- 模型 / 插件 / 提示词版本更新前必须跑一次全量回归

#### 线上评估

- **影子流量**：新版本接收真实流量副本但不返回给用户，用于无风险比对
- **A/B 分流**：按租户 / 用户 / 百分比分流，至少包含"旧版对照组"和"新版实验组"
- **实时指标**：首 Token 时间、完成时间、用户点踩率、未命中引用率
- **自动止损**：指标恶化超过阈值时自动回滚（见 16.3）

#### 错误归因

每次失败请求应自动归因到具体环节：

- 意图识别错误（分类错误、实体缺失）
- 规划失败（步骤不合理、工具选择错误）
- 检索失败（召回低、排序差）
- 插件错误（超时、返回异常、参数不合法）
- 模型错误（胡编、拒答、格式违规）
- 权限拦截（可接受，不计入质量问题）

#### 人类标注回流

- 用户点踩 / 反馈样本自动进入"待标注池"
- 运营人员定期标注后，反哺：①评估集扩充 ②插件排序调整 ③提示词改进候选

### 7.10 记忆与上下文治理模块

#### 功能描述

管理用户、会话、租户、业务包多个作用域下的记忆数据，保障记忆的可用、可控、可被遗忘。

#### 记忆分类

| 类型 | 作用域 | 写入方式 | 典型内容 |
|------|-------|---------|---------|
| 短期会话记忆 | 单次会话 | 自动 | 本次对话历史、临时上下文 |
| 长期用户记忆 | 跨会话 / 单用户 | 需置信度阈值或用户确认 | 用户偏好、角色、常用项 |
| 语义知识记忆 | 租户级 | 运营审核 | 稳定事实、企业术语 |
| 业务状态记忆 | 业务包级 | 插件主动写入 | 最近操作的业务对象上下文 |

#### 关键能力

- **TTL 与过期**：每条记忆强制携带生命周期（如 30 天 / 90 天 / 永久），到期自动归档
- **置信度衰减**：长期未被复用的记忆分数逐步下降，低于阈值后不再召回
- **冲突处理**：新事实与旧记忆冲突时，按时间 + 来源置信度决策，保留旧版本作为历史
- **来源追踪**：每条记忆可回溯到写入来源（用户确认 / 插件回写 / 对话抽取）
- **用户可见可控**：用户可在"我的记忆"页查看、编辑、导出、删除自己的长期记忆
- **被遗忘权**：删除用户时，必须级联清理该用户在记忆、日志、缓存中的 PII

#### 记忆规则

- 未经用户显式确认的信息默认仅作为短期记忆
- 敏感字段（身份证、手机号、薪资等）不得直接写入长期记忆
- 跨租户绝对禁止记忆共享

### 7.11 安全与内容治理模块

#### 功能描述

防御 Prompt 层、数据层、输出层的安全风险，满足企业合规要求。

#### 输入侧防护

- **Prompt Injection 防护**：检测并拦截"忽略上文指令""切换角色""执行系统命令"等模式
- **敏感输入识别**：识别用户主动输入的 PII / 密级信息，按租户策略提示或自动脱敏
- **工具滥用检测**：识别诱导 Agent 跨越权限边界的输入

#### 数据侧防护（DLP）

- **数据分级**：每个知识源、数据源需打上密级标签（公开 / 内部 / 机密 / 绝密）
- **最小权限召回**：检索严格按用户可见密级过滤
- **跨租户隔离**：向量检索、缓存、记忆必须按租户硬隔离（见 10.4）
- **出域控制**：标注为"不可外发"的数据禁止进入外部模型接口

#### 输出侧防护

- **敏感信息脱敏**：默认对输出中的身份证、手机号、银行卡等做模式化脱敏
- **有害内容审查**：调用内容安全服务，拦截违规、违法、歧视内容
- **幻觉兜底**：对无引用支撑的事实性陈述，提示"此结论无知识库依据"
- **攻击性输出拦截**：防止被诱导输出攻击性脚本、钓鱼话术

#### 合规能力

- 支持数据留存期配置（默认 90 天审计日志 / 可按合规需求延长）
- 支持数据主体访问请求（DSAR）：查询、导出、删除
- 支持跨境数据流转开关（按租户地域）
- 定期红队 / 对抗测试纳入发布流程

### 7.12 多模型路由与降级模块

#### 功能描述

屏蔽底层模型差异，按任务类型、租户策略、成本预算调度到合适的模型。

#### 功能点

- **模型抽象层**：统一的 `Chat / Embedding / Rerank / Vision` 接口，底层可切换
- **按任务路由**：规划、抽取、生成、改写、评分等任务可配不同模型
- **按租户路由**：不同租户可绑定不同主模型（考虑合规与成本）
- **成本敏感路由**：低风险任务走轻量模型，复杂任务走强模型
- **主备降级**：主模型限流 / 不可用时自动切到备模型，并在审计中标注
- **模型版本管理**：支持灰度切换、影子评估、一键回滚

#### 非功能要求

- 切换模型不应导致业务包逻辑改动
- 同一请求在重试 / 降级过程中保持 Trace 一致性
- 模型侧错误（超限、内容审查）需被翻译为统一错误码

### 7.13 开发者体验与工具链模块

#### 功能描述

为插件开发者、业务包集成方提供一条"从 0 到上线"的顺滑链路，是平台能否被广泛接入的关键。

#### 必备工具

- **插件脚手架 CLI**：一键生成插件骨架、capability schema 模板、测试用例模板
- **本地调试 SDK**：在本地即可调用平台 Planner、检索、模型，无需部署
- **沙箱租户**：开发者专用环境，数据隔离、调用配额独立
- **契约 Mock 工具**：根据 schema 自动生成 mock 响应，便于前后端并行
- **回归测试框架**：提供标准断言（字段存在、schema 合规、side_effect 声明一致）
- **在线 Playground**：网页端试跑插件，查看完整 Trace、Token 消耗、审计记录
- **文档中心**：协议文档、示例业务包源码、常见接入 FAQ

#### 插件提交流程

1. 开发者在沙箱租户提交插件包
2. 平台自动执行契约校验、回归测试、安全扫描
3. 平台管理员审核通过后进入企业插件仓库
4. 业务管理员按需启用到生产租户

## 8. 核心流程设计

### 8.1 标准查询流程

```text
用户输入
 -> 意图识别
 -> 读取上下文
 -> 候选能力筛选
 -> Planner生成任务计划
 -> 契约校验
 -> 权限校验
 -> 检索/插件执行
 -> 结果整合
 -> 返回答案与依据
 -> 审计与评估回写
```

### 8.2 受控写操作流程

```text
用户发起请求
 -> 风险识别
 -> 权限校验
 -> 生成执行草稿
 -> 用户确认 / 审批
 -> 调用外部系统
 -> 返回结果
 -> 写入审计日志
```

### 8.3 行业包接入流程

```text
定义业务包
 -> 编写插件与Capability Spec
 -> 接入知识源和业务系统
 -> 配置权限和评估规则
 -> 测试与灰度
 -> 正式发布
```

## 9. 需求明细

### 9.1 功能性需求

#### FR-001 统一对话入口

- 系统应提供统一工作台入口
- 系统应支持多轮对话和会话历史
- 系统应支持结果中的引用与依据展示

#### FR-002 任务编排

- 系统应支持根据意图生成任务计划
- 系统应支持多任务依赖分析与执行顺序控制
- 系统应支持失败回退与重试策略

#### FR-003 插件能力调用

- 系统应支持按能力契约调用插件
- 系统应支持插件健康检查和异常隔离
- 系统应支持插件版本管理

#### FR-004 能力契约校验

- 系统应支持定义输入/输出 schema
- 系统应在执行前校验输入
- 系统应在执行后校验输出

#### FR-005 知识与数据检索

- 系统应支持接入文档型知识源与结构化数据源
- 系统应支持检索结果返回引用信息
- 系统应支持按租户和权限过滤检索结果

#### FR-006 权限与审批

- 系统应支持按租户、角色、权限控制能力调用
- 系统应支持高风险操作审批
- 系统应支持审批记录可审计

#### FR-007 审计与监控

- 系统应记录请求、计划、执行链路与结果摘要
- 系统应支持按插件、租户、场景查看运行情况

#### FR-008 业务包管理

- 系统应支持业务包级别启停和版本管理
- 系统应支持为不同租户启用不同业务包

#### FR-009 记忆治理

- 系统应支持多作用域记忆（会话 / 用户 / 租户 / 业务包）
- 系统应支持记忆 TTL、置信度衰减、冲突处理
- 系统应提供"我的记忆"页面供用户查看、编辑、删除
- 系统应在用户被删除时级联清理其 PII

#### FR-010 安全与内容治理

- 系统应具备 Prompt Injection 检测与拦截能力
- 系统应支持数据分级，并按密级过滤检索
- 系统应对输出做敏感信息脱敏与内容合规审查
- 系统应记录所有被安全策略拦截的请求

#### FR-011 配额与限流

- 系统应支持租户级 QPS、日 Token、月成本预算
- 系统应支持 capability 维度的调用频次与熔断
- 配额触达阈值时应告警，超限时应优雅降级

#### FR-012 多模型路由

- 系统应提供统一的模型抽象层
- 系统应支持按任务、租户、成本维度配置路由策略
- 系统应支持主备模型降级与灰度切换

#### FR-013 开发者工具链

- 系统应提供插件脚手架 CLI 与本地调试 SDK
- 系统应提供沙箱租户供插件开发与测试
- 系统应提供插件提交、自动化校验、审核发布的完整流程

### 9.2 非功能性需求

#### NFR-001 性能

按任务类型分档：

| 场景类型 | 首 Token 时延 (P50) | 完成时延 (P50) | 完成时延 (P95) |
|---------|--------------------|----------------|----------------|
| 问答 / FAQ | ≤ 1.5s | ≤ 5s | ≤ 10s |
| 数据查询 | ≤ 2s | ≤ 8s | ≤ 15s |
| 复杂分析（多插件） | ≤ 3s | ≤ 15s | ≤ 30s |
| 受控执行（不含人工审批时长） | ≤ 3s | ≤ 20s | ≤ 40s |

其他性能约束：

- 峰值并发请求按租户规模容量评估，单租户不得影响其他租户响应
- 流式输出应在 2s 内返回首个可见 Token

#### NFR-002 可用性

- 平台月度可用性不低于 99.5%
- 单个插件故障不得拖垮整个平台（故障隔离）
- 模型厂商故障时应在 30s 内完成主备切换

#### NFR-003 安全性

- 支持 SSO、多租户隔离、敏感字段脱敏、越权拦截
- 支持 Prompt Injection 防护与输出内容审查（见 7.11）
- 审计日志不可篡改，默认保留 180 天（可按合规延长）
- 支持数据主体访问请求（DSAR）

#### NFR-004 可扩展性

- 新业务包接入不应修改核心框架
- 新知识源、新模型、新插件应支持标准化接入
- 平台自有插件契约应可导出为 MCP 规范，支持与外部 Agent 生态互通

#### NFR-005 可观测性

- 支持日志、指标、Trace 三件套
- **全链路 Trace ID** 贯穿：意图识别 → 规划 → 检索 → 插件 → 模型 → 审计
- 每个节点必须记录耗时、状态、输入输出摘要、Token 消耗
- 支持按租户、业务包、插件、模型多维过滤
- 支持采样策略：错误请求 100% 采样、正常请求按比例采样
- 用户反馈（点赞 / 点踩 / 文本反馈）应与 Trace 关联

#### NFR-006 数据保留与灾备

- 会话原文默认保留 30 天，可按租户合规要求调整
- 审计日志默认保留 180 天
- 每日异地备份，RPO ≤ 1 小时，RTO ≤ 4 小时

### 9.3 用户故事与验收描述

#### US-001 终端用户发起问答

作为业务使用者，我希望在统一对话页中直接提问，并看到带依据的回答，从而快速获得结论且知道答案来自哪里。

验收标准：

- 用户进入对话页后，可直接输入自然语言问题
- 若命中知识检索，答案区必须展示引用来源
- 若命中业务插件，答案区必须展示数据来源摘要
- 回答失败时，系统需展示可理解的失败说明，而不是技术异常堆栈

#### US-002 用户发起受控写操作

作为业务使用者，我希望在平台中生成流程草稿并在确认后提交，而不是系统直接替我执行高风险动作。

验收标准：

- 当请求包含写操作时，系统需进入草稿确认态
- 草稿页需展示待执行对象、动作、参数摘要和风险级别
- 用户未确认前，不得写入外部系统
- 审批完成后，系统需返回执行结果和审计编号

#### US-003 业务管理员启用业务包

作为业务管理员，我希望为本租户启用或停用某个业务包，以便控制哪些能力对终端用户可见。

验收标准：

- 管理后台可查看业务包列表、版本和状态
- 业务管理员仅能操作自己有权限范围内的业务包
- 启停操作后，对话页的可用能力入口应同步刷新

#### US-004 平台管理员查看运行异常

作为平台管理员，我希望查看某个请求的完整执行链路，从而快速定位问题是出在模型、检索还是插件。

验收标准：

- 支持通过请求 ID 检索执行链路
- 可看到意图识别、规划、检索、插件执行的关键节点
- 每个节点应展示耗时、状态和错误摘要

### 9.4 权限矩阵

| 功能 | 业务使用者 | 业务管理员 | 平台管理员 | 审计人员 |
|------|------|------|------|------|
| 发起普通查询 | 是 | 是 | 是 | 否 |
| 发起受控写操作 | 按权限 | 按权限 | 是 | 否 |
| 查看个人会话历史 | 是 | 是 | 是 | 脱敏后可见 |
| 查看他人会话历史 | 否 | 否 | 是（审计场景） | 脱敏后可见 |
| 查看租户级运营数据 | 否 | 是 | 是 | 是 |
| 启停业务包 | 否 | 按租户授权 | 是 | 否 |
| 管理插件版本 | 否 | 否 | 是 | 否 |
| 查看审计日志 | 否 | 部分可见 | 是 | 是 |
| 配置模型与检索参数 | 否 | 否 | 是 | 否 |
| 配置租户配额与预算 | 否 | 查看 | 是 | 否 |
| 导出 / 删除个人记忆 | 本人 | 否 | 响应 DSAR | 否 |

说明：审计人员对会话与记忆的查看默认走"脱敏视图"（PII 掩码），仅在触发特定合规流程时可申请原文，全过程需二次审批并留痕。

## 10. 业务规则

### 10.1 平台规则

- 平台只负责编排与治理，不固化行业业务逻辑
- 所有业务逻辑应优先封装在业务包/插件中
- 平台层输出需明确来源，不允许无依据生成关键事实

### 10.2 插件规则

- 所有插件能力必须声明 capability schema（见 7.6）
- 写操作必须声明副作用级别（none / read / write / irreversible）和权限要求
- "高风险"按 10.5 风险分级矩阵统一判定，不由插件开发者自行认定
- high / critical 级动作默认需要审批或人工确认，不得通过配置单方面降级

### 10.3 记忆规则

- 短期会话记忆自动写入，仅在当前会话上下文内有效
- 长期记忆需满足以下任一条件方可写入：①来源为可信插件回写 ②用户显式确认 ③规则引擎判定为稳定事实
- 每条记忆必须携带：来源、写入时间、置信度、TTL、作用域
- 敏感字段（身份证、手机号、银行卡、薪资等）默认不写入长期记忆
- 用户可在"我的记忆"页查看、编辑、导出、删除自己的记忆
- 用户账号被删除时，相关长期记忆必须在 T+7 天内级联清理

### 10.4 租户隔离等级

平台定义以下隔离等级，新业务包 / 新数据源接入时必须声明其适用等级：

| 维度 | 最低等级 | 说明 |
|------|---------|------|
| 会话数据 | 物理隔离或逻辑强隔离 | 禁止跨租户查询 |
| 向量库 / 检索索引 | 按租户分 index / namespace | 召回阶段强过滤 |
| 长期记忆 | 按租户分库或强隔离 | 绝对禁止跨租户共享 |
| 缓存 | 按租户前缀 Key | 禁止通用缓存命中跨租户 |
| 模型调用 | 按租户路由 | 敏感租户可绑定专属模型 |
| 日志 / 审计 | 按租户标签 | 查询时强过滤 |
| 插件运行环境 | 按业务包 / 租户沙箱 | 敏感插件可申请独立实例 |

硬规则：**任何一处跨租户泄露都视为 P0 级安全事件，必须立即熔断该能力并启动应急响应。**

### 10.5 风险分级矩阵

所有 capability、所有执行动作必须在契约中声明风险等级：

| 等级 | 判定依据 | 默认策略 |
|------|---------|---------|
| low | 只读、幂等、无个人数据 | 直接执行 |
| medium | 只读但涉及个人数据 / 跨模块聚合 | 留痕 + 脱敏输出 |
| high | 写操作、可逆、影响单个业务对象 | 强制走草稿 / 审批（L2+） |
| critical | 不可逆、批量、涉及资金 / 权限 | 强制二级审批 + 人工确认 + 熔断保护 |

- 对话消息卡片组件

- 审批状态组件

- 执行链路时间线组件

### 12.4 开发者体验（DX）实现建议

面向"集成开发者"用户的系统能力建议：

- **脚手架 CLI**：`agent-platform init <plugin-name>`，产出标准目录结构、schema 示例、测试用例

- **开发者文档中心**：快速上手、协议规范、API 参考、FAQ、治理规则

## 14. 指标设计

### 14.1 业务指标

- 平台活跃用户数
- 问题解决率
- 平均任务完成时长
- 业务包使用覆盖率
- 审批前草稿采用率

### 14.2 平台指标

- intent accuracy
- valid plan rate
- task success rate
- retrieval recall@k
- plugin error rate
- approval intercept rate

### 14.3 运营指标

- 单请求平均成本
- 各业务包调用量
- 各租户资源消耗
- 模型与检索成本占比
- 租户月度成本 vs 预算达成率

### 14.4 北极星指标与指标关联

**北极星指标**：活跃用户每周解决的任务数（Tasks Resolved Per Active User / Week）

围绕北极星指标的支撑关系：

- 活跃用户数 ← 新用户留存率、能力发现效率、用户满意度
- 任务解决率 ← intent accuracy × valid plan rate × retrieval recall × plugin success rate
- 用户满意度 ← 点赞率、点踩率、重复提问率、主动放弃率

### 14.5 反馈闭环

- 用户点赞 / 点踩 / 文本反馈统一进入"反馈池"
- 反馈样本按类型自动路由：
  - 意图错判 → 意图模型 / 规划器评估集
  - 引用错误 → 知识质量评分
  - 插件错误 → 插件健康度与排序
  - 回答不准 → 提示词 / 模型路由候选
- 每周生成"反馈驱动改进"报告，指出 Top N 待优化项

## 16. 风险与约束

### 16.1 主要风险

- 插件协议不清晰会导致扩展成本快速升高
- 不同业务包质量不均会影响平台口碑
- 权限与审计设计不足会带来合规风险
- 行业包需求过早下沉到底座，会破坏平台通用性

### 16.2 应对策略

- 平台与业务包边界前置定义
- 通过 capability schema 强约束接入质量
- 通过评估回归与灰度机制控制上线风险
- 高风险能力统一纳入审批和审计框架

### 16.3 灰度与回滚策略

任何插件、模型、提示词、业务包发布都必须具备可回滚路径。

#### 分流维度

- **按租户**：先在 1–2 个试点租户开启
- **按用户**：租户内白名单 / 百分比灰度
- **按场景**：仅对低风险场景生效（如问答先于写操作）
- **影子流量**：与生产流量并行不返回用户，仅用于评估

#### 自动回滚触发条件

| 指标 | 阈值示例 | 回滚动作 |
|------|---------|---------|
| 任务失败率 | 10 分钟内 > 基线 + 5pp | 立即切回上一版本 |
| P95 时延 | 10 分钟内 > 基线 × 1.5 | 切回上一版本 |
| 点踩率 | 1 小时内 > 基线 + 10pp | 告警 + 人工确认 |
| 审批拒绝率 | 1 小时内 > 基线 + 20pp | 告警 + 暂停发布 |
| 成本 | 单小时 > 预算 × 2 | 降级到备用模型 |
| 安全事件 | 任意一起 P0 级 | 立即熔断 + 上报 |

#### 回滚保障

- 每个版本必须保留最近 3 个可回滚版本
- 回滚不应丢失用户会话上下文
- 回滚动作必须记入审计日志

## 17. 验收标准

### 17.1 P0a 验收标准（基础底座）

- 平台可通过统一入口完成知识问答与业务查询（L1 能力）

- 任一插件故障不影响整体平台可用性

- 平台支持至少 2 个业务包独立接入
- 平台具备离线评估、线上 A/B 与灰度发布能力

- 开发者工具链可支撑插件从接入到发布全流程

### 17.4 P2 验收标准

- 平台支持受控执行与审批链路（L2 / L3）
- 平台支持租户级业务包配置、配额与隔离
- 平台具备运营看板和平台治理能力
