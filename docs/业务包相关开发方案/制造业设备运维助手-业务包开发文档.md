## 2. 业务背景与场景

### 2.1 工厂现状

离散制造车间一线管理者每天处理大量"信息散落 + 多系统切换"的事务：

- 设备报警 → 翻 PDF 手册、查 EAM 历史、联系维修工、走停机审批
- 备件领用 → 在 ERP 翻库存、找货位、起领料单、找主管签字
- 换型/点检 → 找 SOP、查上次记录、排人、报工时
- 故障复盘 → 在多个系统拷数据，做表格

信息系统至少包括：MES、EAM、备件 ERP、排班/HR、企业微信/对讲机、纸质/电子 SOP 库。

### 2.2 痛点

1. **响应慢**：故障到派工平均 12-25 分钟，多数时间花在查信息和找人。
2. **经验不传承**：老师傅脑子里的处置流程没沉淀，新人独立处置慢、易出错。
3. **审批跑断腿**：紧急停机要主任签字，主任不在场就拖。
4. **复盘成本高**：故障后做改进报告要人工拼数据。
5. **安全风险**：操作不规范、备件用错、特种作业证过期派工等，都靠人工把关。

### 2.3 业务包目标

- 把"老师傅的处置流程 + 必须遵守的合规边界"显式编码为 Agent 行为
- 把"5 个系统切换"压缩为"一句话提问 + 一次确认"
- 故障响应时间从 12-25 分钟降到 3-5 分钟
- 关键安全规则（断电、泄压、证件、连续工时）由插件强制拦截，不依赖模型自由发挥

### 2.4 适用边界

适用：

- 离散制造产线，设备数量 ≥ 50 台
- 已上线 EAM、ERP（或可对接）
- 至少有班长 / 维修工 / 工程师 / 主任四类角色

暂不适用：

- 流程行业（化工、冶金）—— 安全等级和审批模型差异大，需要专门业务包
- 设备资产 < 50 台、无 EAM 的小厂 —— 业务系统不齐，价值不显著

## 4. 核心场景剧本

剧本 = 意图 + 推荐 capability 序列 + 草稿动作 + 引用要求。剧本是业务包的"骨架"，Planner 在该业务包上下文中优先按剧本规划。

### 4.1 剧本 A：故障处置（最高频）

**触发**：用户说"X 号设备报 E-XXX，怎么处理"或 MES 报警事件推到 Agent。

**Planner 默认序列**：

```text
1. eam.equipment.profile.query(equipment_id)        # 设备身份
2. knowledge.search(query="故障码 E-XXX 处置")       # 从手册和历史工单召回
3. eam.workorder.history.query(equipment_id, fault_code, last_n=5)  # 历史
4. erp.spare.stock.query(part_no_list)              # 推断需要的备件并查库存
5. shift.skill.match(skill="液压维修", oncall=true)  # 找人
6. 并行生成三张草稿：
   - eam.workorder.draft.create
   - erp.spare.requisition.draft
   - shift.dispatch.draft
7. 返回结构化草稿包 + 一键确认
```

**响应必须包含**：

- 故障码语义解释、安全注意事项（强制 SOP 引用）
- 历史最近一次同类故障的处置摘要 + 工单链接
- 备件型号、库存、货位、替代件提示
- 推荐维修工（姓名、工号、当前工单数、对讲机频道）
- 三张草稿摘要（停机时长、领料数量、派工通知文案）

**审计要点**：每张草稿带 `incident_id`（业务幂等键），三张草稿同源，确认时同步落单。

### 4.2 剧本 B：备件申领

**触发**："给我领 5 个 XX-501"或基于剧本 A 衍生。

**Planner 序列**：

```text
1. erp.spare.stock.query(part_no)
2. 阈值判断（金额 / 数量 / 是否安全库存以下）
3. erp.spare.requisition.draft 或 erp.spare.purchase.draft
4. 命中审批策略 → 生成审批单
```

**关键规则**：

- 金额 < 500 元：班长确认即下发
- 500-5000 元：车间主任审批
- ≥ 5000 元 或低于安全库存：走采购申请流程，财务系统介入

### 4.3 剧本 C：换型 / 点检

**触发**："3 号线下午要换 A 型→B 型，怎么准备"

**Planner 序列**：

```text
1. knowledge.search(query="A→B 换型 SOP")
2. eam.equipment.status.query
3. erp.spare.stock.query（换型耗材）
4. shift.skill.match（持证换型工）
5. 生成换型工单草稿 + 物料齐套清单
```

### 4.4 剧本 D：故障复盘

**触发**："上周 3 号冲压机三次报警，做个复盘"

**Planner 序列**：

```text
1. eam.workorder.history.query(equipment_id, time_range, fault_pattern)
2. 聚合：故障码分布、平均 MTTR、共性原因
3. knowledge.search 找相似案例
4. 生成复盘报告草稿（结构化，含改进建议）
```

注意：此剧本只读，但输出会作为 `workorder_kb` 知识沉淀候选，需经设备工程师确认才入库。

### 4.5 剧本 E：SOP / 知识问答

**触发**："液压系统泄压怎么操作"

**Planner 序列**：

```text
1. knowledge.search（绑定 sop_library + safety_regulation）
2. 直接生成回答，强制引用 SOP 章节与安全规程
3. 不触发任何写操作
```

**红线**：回答中如检测到安全关键操作（断电、泄压、登高、有限空间），必须在前置位置插入安全提示。该规则由 OutputGuard 强制，不依赖模型。

## 5. 角色权限与审批矩阵

### 5.1 角色定义

| 角色 | 工厂语义 | 平台 Scope |
|------|------|------|
| `operator` | 操作工 | `mfg.equipment.read`, `mfg.sop.read` |
| `shift_lead` | 班长 | + `mfg.workorder.draft`, `mfg.spare.requisition.draft`, `mfg.dispatch.draft` |
| `maintenance` | 维修工 | + `mfg.workorder.execute`, `mfg.spare.requisition.execute_self` |

| `workshop_director` | 车间主任 | + `mfg.approval.workshop` |
| `purchaser` | 采购员 | + `mfg.purchase.execute`, `mfg.spare.price.read_full` |
| `safety_officer` | 安全员 | + `mfg.safety.audit` |

### 5.2 审批矩阵

| 动作 | 班长 | 维修工 | 工程师 | 主任 | 采购 |
|------|------|------|------|------|------|
| 查设备/历史/SOP | ✓ | ✓ | ✓ | ✓ | - |
| 起草工单 | ✓ | ✓(自己的) | ✓ | ✓ | - |
| 关闭工单 | 确认 | 起草 | 复核 | 复核 | - |
| 领料 < 500 | ✓ | ✓ | ✓ | ✓ | - |
| 领料 500-5000 | 起草 | - | 起草 | **审批** | - |
| 领料 ≥ 5000 / 采购 | 起草 | - | 起草 | 复核 | **审批** |
| 派工 | ✓ | - | ✓ | ✓ | - |
| 加班申请 | 起草 | - | - | **审批** | - |
| 看备件单价 | 区间脱敏 | 区间脱敏 | 完整 | 完整 | 完整 |

### 5.3 数据可见性

- 排班插件**永远不返回**工资、绩效、考勤异常字段
- ERP 单价对非采购/工程师/主任做"区间脱敏"（如显示"100-150 元"）
- 跨车间数据隔离：班长只能看到自己车间的设备和人员

### 6.3 插件内部分层

每个插件内部按"handler → mapping → client"三层组织：

| 层 | 职责 | 不允许做 |
|----|------|------|
| handler | 接收 capability payload，编排业务流程 | 直接拼 SQL、直接调外部 HTTP |
| mapping | 业务字段 ↔ 外部系统字段双向转换 | 包含业务规则 |
| client | 调外部系统、重试、超时、熔断 | 关心业务字段含义 |

### 6.4 幂等与补偿

所有 `write` / `write_draft` capability 必须支持：

- `incident_id` 或 `idempotency_key` 业务幂等键
- 在 `execution_record` 表登记 `(plugin, capability, idempotency_key) → result`
- 同键二次调用直接返回首次结果，不再调外部系统

补偿场景（举例）：

- 三张草稿同步落单时，第三张失败 → 前两张自动作废 + 通知班长
- EAM 工单已下发但 ERP 领料失败 → 工单挂"等料"状态，不可关闭

### 6.5 业务级错误翻译

插件返回的错误必须是业务级，不暴露底层异常：

| 内部异常 | 对外错误码 | 对外文案 |
|------|------|------|
| EAM 5xx | `EAM_UNAVAILABLE` | "设备系统暂时不可用，已切换为只读模式" |
| 备件不存在 | `PART_NOT_FOUND` | "未找到该备件，请确认料号" |
| 库存不足 | `STOCK_INSUFFICIENT` | "XX-501 当前 0 件，预计 3 天到货；可选替代件 XX-502" |
| 证件过期 | `CERT_EXPIRED` | "候选人 XX 的特种作业证已过期，已自动剔除" |
| 跨车间访问 | `TENANT_ISOLATION_VIOLATION` | "您没有该车间的查看权限" |

## 7. 知识源设计

### 7.1 知识源清单

| knowledge_source | 内容 | 更新频率 | 责任人 |
|------|------|------|------|
| `equipment_manual` | 设备手册、故障码表、电气/液压图 | 设备引入时 + 大修后 | 设备工程师 |
| `sop_library` | 操作 SOP、点检 SOP、换型 SOP | 季度评审 | 工艺工程师 |
| `safety_regulation` | 安全规程、应急预案、特种作业要求 | 年度 + 法规变更 | 安全员 |
| `workorder_kb` | 历史工单沉淀的处置经验 | 周度（工程师审核） | 设备工程师 |

### 7.2 切片元数据扩展原则

**不在 `knowledge_chunk` 表上为每个行业加列**。每加一个行业就改表结构会让平台核心表越来越脏，跨行业查询和 ORM 映射都会失控。采用三层结构：

```text
平台固定列（所有行业共用）
  + attributes JSONB（按业务包 schema 自由扩展）
  + 热点字段提升（被频繁过滤的字段升级为生成列 / 物化列 + 索引）
```

**层 1：平台固定列**（已在通用架构 §6.4 定义，不受业务包影响）

`tenant_id / source_id / document_id / chunk_index / version / classification / updated_at / locator / status / content / embedding`。

**层 2：业务包 attributes（JSONB）**

`knowledge_chunk` 增加唯一一列扩展字段：

```sql
ALTER TABLE knowledge_chunk
  ADD COLUMN IF NOT EXISTS attributes jsonb NOT NULL DEFAULT '{}'::jsonb;

CREATE INDEX IF NOT EXISTS idx_kc_attributes_gin
  ON knowledge_chunk USING gin (attributes jsonb_path_ops);
```

每个业务包在 `knowledge-bindings.yaml` 中**自带 schema** 声明它要往 `attributes` 里塞什么、每个字段类型与是否参与召回过滤：

```yaml
# mfg_equipment_ops 的扩展 schema
chunk_attributes_schema:
  equipment_model:
    type: string
    indexed: hot           # 标记为热点字段，平台为其建生成列+索引
    filter: in
  fault_code_tags:
    type: array<string>
    indexed: hot
    filter: array_contains
  safety_critical:
    type: boolean
    indexed: warm          # GIN 索引足够
    filter: eq
  sop_step:
    type: string
    indexed: cold          # 仅展示，不过滤
  effective_date:
    type: date
    indexed: hot
    filter: lte
```

入库流水线（通用架构 §6.4）按业务包注册的 schema 校验 `attributes`，校验失败的 chunk 不进入 `published`。这样平台代码完全不感知 `equipment_model` 这种行业字段，只感知"业务包声明了一个 string 类型、需热点索引的过滤字段"。

**层 3：热点字段提升**

被高频用作过滤条件的字段不能只靠 JSONB GIN 索引（选择性差时性能不够）。平台对 `indexed: hot` 字段自动生成 PostgreSQL **generated column + 索引**：

```sql
-- 由平台根据业务包 schema 自动生成，不需要业务包开发者写
ALTER TABLE knowledge_chunk
  ADD COLUMN IF NOT EXISTS attr_equipment_model varchar(64)
    GENERATED ALWAYS AS (attributes->>'equipment_model') STORED;

CREATE INDEX IF NOT EXISTS idx_kc_attr_equipment_model
  ON knowledge_chunk(tenant_id, attr_equipment_model);
```

> 为什么不用纯 JSONB？JSONB GIN 在低基数 + 范围过滤场景退化明显（如 `effective_date <= now()`）。生成列保证 BTree 索引可用，又不影响逻辑模型。
>
> 为什么不用 EAV 子表？查询要 N 次 JOIN，跨行业组合过滤性能更差。

**升级机制**：业务包发布或变更时，`PackageLoader` 比对 schema diff，自动产出 ALTER 迁移脚本（仅 ADD COLUMN / CREATE INDEX，不允许 DROP / ALTER TYPE，避免动到别的业务包）。回滚时只把生成列设为不可见，不删数据。

### 7.3 检索过滤规则

`knowledge.search` 不直接拼 `equipment_model = ?` 这种行业 SQL，而是平台层的 **AttributeFilterCompiler** 把"业务包过滤意图"翻译成 SQL：

业务包侧声明（`knowledge-bindings.yaml`）：

```yaml
retrieval:
  filter_template:
    fault_handling:
      - field: equipment_model
        op: in
        from_context: equipment_model_with_compatible
      - field: effective_date
        op: lte
        from_context: now
      - any_of:
          - field: fault_code_tags
            op: array_contains
            from_context: fault_code
          - field: safety_critical
            op: eq
            value: true
```

平台编译产出的实际 SQL（伪）：

```sql
WHERE tenant_id = :tenant
  AND status = 'published'
  AND attr_equipment_model = ANY(:models)        -- 来自生成列
  AND (attributes @> jsonb_build_object('fault_code_tags', :fault_code_array)
       OR (attributes->>'safety_critical')::boolean = true)
  AND (attributes->>'effective_date')::date <= now()
```

热点字段走生成列 BTree，长尾字段走 JSONB GIN，业务包开发者只写 yaml，不写 SQL。新行业接入时只需要：

1. 在自己的 `knowledge-bindings.yaml` 写 schema 与 filter_template
2. 平台自动生成索引
3. 平台核心表零变更

### 7.4 引用展示要求

回答中每条引用展示：

- 来源类型（手册/SOP/安全规程/历史工单）
- 文档名 + 章节 / 步骤号
- 版本与生效日期
- "查看原文"链接（跳转到文档定位锚点）

历史工单引用额外展示：处置人、当时停机时长、是否解决。

## 9. 数据模型扩展

业务包不引入"行业专属表"，复用通用架构 §12 的核心表，通过元数据扩展承载行业字段。

### 9.1 平台核心表的扩展原则

**核心表只加一次通用扩展位，不为任何具体行业改 schema**。详见 §7.2。涉及通用平台的迁移只有两条，且对所有业务包共享：

```sql
ALTER TABLE knowledge_chunk
  ADD COLUMN IF NOT EXISTS attributes jsonb NOT NULL DEFAULT '{}'::jsonb;

CREATE INDEX IF NOT EXISTS idx_kc_attributes_gin
  ON knowledge_chunk USING gin (attributes jsonb_path_ops);
```

行业字段（如 `equipment_model`、`fault_code_tags`）由 `PackageLoader` 在装配时按业务包 schema **自动生成 generated column + 索引**，命名规则 `attr_<field>`，迁移脚本由平台产出，业务包开发者不直接写 DDL。

### 9.1.1 新业务包接入的零侵入清单

| 接入项 | 是否改平台代码 / 表结构 |
|------|------|
| 增加扩展属性字段 | 否，写在业务包 yaml 的 `chunk_attributes_schema` |
| 扩展属性建索引 | 否，平台按 `indexed: hot/warm/cold` 自动生成 |
| 新增检索过滤规则 | 否，写在业务包 yaml 的 `filter_template` |
| 新增 capability | 否，插件包注册即可 |
| 新增剧本意图 | 否，业务包 `capability-map.yaml` 声明 |
| 新增审批阈值 | 否，业务包 `approval_policy.yaml` 声明 |

只有当业务包需要**全新业务对象**（如本业务包的 `mfg_incident`，承载故障事件聚合）时，才在自己的命名空间下加表，**不动 `knowledge_chunk` / `conversation` / `request_trace` 等核心表**。

### 9.2 业务包绑定表（来自通用架构 §8.7）

业务包级的"哪个租户启用了 1.0.0"由通用 `tenant_package_release` 承载，无需新建表。`mfg_equipment_ops` 在该表中体现为：

```text

business_package_version = 1.0.0
tenant_id = t_factory_001
status = released
gray_strategy = workshop_whitelist
gray_payload = {"workshops": ["W01"]}
```

### 9.3 业务事件表（建议新增）

为支撑剧本 A 的"三张草稿同源"和复盘剧本：

```sql
CREATE TABLE IF NOT EXISTS mfg_incident (
```

## 10. 运行时装配流程

### 10.1 装配时机

支持两种装配时机：

1. **启动时**：平台启动加载所有 `released` 状态的业务包，缓存能力图谱。
2. **请求时**：根据 `tenant_id + context.business_package` 拼接最终能力视图。

推荐"启动时全量加载 + 请求时按租户过滤"。

### 10.2 装配步骤

```text
1. PackageLoader 读取 packages/mfg_equipment_ops/package.yaml
2. 解析 plugins[] -> 校验插件包存在且版本匹配
3. 解析 capability-map.yaml -> 形成 capability 白名单
4. 解析 knowledge-bindings.yaml -> 形成知识源过滤器
5. 解析 approval_policy.yaml -> 注册到 ApprovalService
6. 装载 prompts/* -> 形成业务包提示词上下文
7. 在 CapabilityRegistry 中标记每个 capability 所属 package
8. 缓存到 PackageRuntimeView (按 package_version 维度)
```

### 10.3 请求级解析

Supervisor 在每次请求开始时：

```text
1. 解析 RequestContext: tenant_id + business_package + user_scopes
2. 查 tenant_package_release: 该租户是否启用了 mfg_equipment_ops 1.0.0
3. 命中灰度规则后，加载 PackageRuntimeView
4. 用 user_scopes 过滤 capability 白名单 (得到本次可用能力)
5. 注入业务包提示词、Planner 默认策略、风险上限
6. 调用意图分类：limit_to = entry_intents
7. Planner 拿到的 capability 池 = intent_to_capabilities[intent] ∩ 用户可用能力
```

### 10.4 与现有代码的衔接

落到当前仓库 `apps/api/src/agent_platform/`：

| 改造点 | 文件 | 动作 |
|------|------|------|
| 业务包加载器 | `runtime/package_loader.py` (新增) | 读取 yaml、缓存视图 |
| Capability 注册 | `runtime/registry.py` | 由硬编码改为按 package 装载 |
| Supervisor | `runtime/chat_service.py` | 注入 business_package 上下文，按剧本规划 |
| 数据模型 | `domain/models.py` + `infrastructure/repositories.py` | 新增 `mfg_incident`、扩展 `knowledge_chunk` |
| 插件 | `plugins/mfg_eam/`, `plugins/mfg_erp_spare/` ... | 新增 |
| 配置 | `packages/mfg_equipment_ops/` (新增) | yaml + prompts + evals |

### 10.5 装配 Trace

每次请求 Trace 步骤增加 `package_assembled`：

```json
{
  "step": "package_assembled",
  "package": "mfg_equipment_ops",
```

## 11. Planner 策略与提示词

### 11.1 默认策略

业务包默认 `plan_execute`：

- 故障处置剧本步骤稳定，适合先规划再执行
- 不使用 ReAct，避免中间 LLM 决策导致漏掉关键步骤
- 安全敏感问题强制走 `direct_answer + 强引用`

### 11.2 系统提示词（节选）

`prompts/system_prompt.txt`：

```text
你是制造业设备运维助手，服务于离散制造工厂的班长、维修工和工程师。

行为准则：
1. 安全第一。涉及断电、泄压、登高、有限空间、动火等操作时，必须先复述
   安全规程要求，再讲操作步骤。安全提示不能省略，不能改写。
2. 术语标准化。使用设备手册和 SOP 中的官方术语，禁止口语化替换
   （如不得把"溢流阀"说成"减压阀"）。
3. 引用强制。任何处置建议必须给出来源（手册章节、SOP 步骤、历史工单号）。
   无来源时直接说"暂无依据，请联系工程师"。
4. 不臆造数据。不知道的库存、人员、单价一律用工具查询，不允许猜测。
5. 写操作只起草。所有工单、领料、派工只生成草稿，由用户确认后才下发。
6. 紧急情况优先派人。危及人身安全的描述出现时，立即建议拨打车间应急电话，
   不要等 Agent 流程跑完。
```

### 11.3 Planner 提示词（节选）

`prompts/planner_prompt.txt`：

```text
你是规划器。根据用户问题判断意图，选择 capability 序列。

可用意图：fault_handling | spare_lookup | spare_requisition |
          dispatch_request | equipment_status | sop_query |
          workorder_close | incident_review

规划规则：
- fault_handling 必须先调 eam.equipment.profile.query 和 knowledge.search，
  再决定是否调备件查询和派工。
- 任何写操作必须放在序列末端，且标记 requires_confirmation=true。
- 同一 incident 的多个写操作必须使用相同 incident_id。
- 步骤数 <= 8。超过则停止规划，提示用户拆分问题。
```

### 11.4 OutputGuard 规则

- `safety_prefix`: 当响应中引用了 `safety_critical = true` 的 chunk，强制在
  回答开头插入"安全提示"段
- `term_check`: 检测响应中是否出现 SOP 词典外的设备术语，命中则降级或
  追加"请以手册术语为准"
- `pii_mask`: 维修工手机号、工号在跨车间展示时脱敏

## 12. 安全、风险与合规

### 12.1 安全红线（OutputGuard 强制）

| 红线 | 触发条件 | 动作 |
|------|------|------|
| 漏说断电/泄压 | 涉及电气/液压维修但响应中无对应安全步骤 | 拒绝输出，要求重生成 |
| 跳过 LOTO | 涉及上锁挂牌作业未提及 | 强制追加 LOTO 说明 |
| 派给无证人员 | 候选人证件过期或不持证 | 插件层即剔除，Planner 不可见 |
| 超连续工时 | 候选人当日已工作 ≥ 12h | 插件剔除并附理由 |
| 跨车间数据 | 响应中出现非用户车间的设备/人员 | 拦截并审计 |

### 12.2 风险等级映射（对应通用架构 §11.4）

| 业务包动作 | 风险 | 处理 |
|------|------|------|
| 知识查询、设备状态查询 | low | 直接执行 |
| 起草工单、起草领料 | medium | 草稿 + 用户确认 |
| 关闭工单、派工通知 | medium | 确认 + 审计 |
| 大额采购、加班申请 | high | 审批 + 双签 |
| 跨车间操作、绕过证件 | critical | 直接拦截 + 安全事件 |

### 12.3 配额建议

| 范围 | 阈值 |
|------|------|
| 单租户每日工单草稿 | 500 |
| 单用户每分钟提问 | 20 |
| 备件采购草稿 / 班次 | 30（防误触发风暴） |
| `notify.broadcast` / 班次 | 5 |

### 12.4 合规与留痕

- 所有写操作 Trace 永久保留，不参与采样丢弃
- 安全红线触发记录单独入 `security_events` 表，月度汇总给安全员
- 涉及人员排班的查询记录保留 90 天，满足劳动法相关追溯

## 14. 灰度发布与运营

### 14.1 灰度路径

```text
1. 沙箱租户验证（2 周）
   - 设备工程师 + 安全员人工跟测每条剧本
2. 单产线灰度（2 周）
   - 选 1 条非关键产线，班长 + 维修工真实使用
   - 监控误派工率、安全红线触发数
3. 单车间灰度（4 周）
   - 扩到整个车间，覆盖所有班次
4. 全厂发布
   - 主任和安全员双签
```

### 14.2 自动回滚触发

- 安全红线漏过 ≥ 1 次（**必须立即回滚 + 复盘**）
- 误派工率 > 10%
- 故障处置剧本失败率 > 5%
- EAM/ERP 写操作失败率 > 2%

### 14.3 运营看板

按业务包维度切分，关键指标：

- 各剧本调用量 / 成功率
- 各 capability 平均时延
- 草稿确认率 / 撤销率
- 审批等待时长
- 安全红线命中分布
- Top 故障码分布（用于反哺知识库）

### 14.4 反馈闭环

- 每个回答附"有用 / 没用 / 错误"按钮
- "错误"反馈进入工程师工作流，确认后：
  - 修知识库 → 重新发布
  - 修 SOP → 同步到 sop_library
  - 修 Planner 提示词 → 进入业务包下一版本
