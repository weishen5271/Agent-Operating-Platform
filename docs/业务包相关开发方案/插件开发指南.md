## 2. 插件的定位与边界

### 2.1 三层分工（重要）

```text
连接细节  →  插件 (plugin)
业务规则  →  业务包 (package yaml)
环境配置  →  租户配置 / 配置中心
```

| 内容 | 归属 |
|------|------|
| 接口地址、协议、鉴权方式 | 插件 |
| 字段名映射（业务字段 ↔ 系统字段） | 插件 |
| 重试、超时、熔断、错误翻译 | 插件 |
| 幂等键计算、补偿 | 插件 |
| 剧本顺序、审批阈值、角色权限 | **业务包** |
| 哪些字段对哪些角色脱敏 | **业务包**（脱敏规则）+ 插件（执行） |
| 服务地址、API Key、租户 ID | **租户配置** |

### 2.2 判别口诀

- 换一家厂商，要改 → 写在插件里
- 换一家厂商不该改 → 写在业务包里
- 同一个业务包不同环境不一样 → 写在租户配置里

### 2.3 插件的硬边界

插件**只能**通过平台 SDK 输出 capability，**不能**：

- ❌ 直接读写平台核心表（`tenant`、`conversation`、`request_trace`、`knowledge_chunk` 等）
- ❌ 绕过 PDP（权限决策点）调用其他插件
- ❌ 自行写审计日志（必须通过 SDK，由平台统一打）
- ❌ 注入自定义中间件、改 Supervisor 行为
- ❌ 自行决定是否需要审批（这是业务包 + 平台治理的事）
- ❌ 在响应里夹带不在 `output_schema` 里的字段

插件的世界里只有：**接收 payload → 调外部系统 → 返回符合 schema 的结果**。

## 3. 插件包结构

### 3.1 标准目录

```text
plugins/<industry>_<system>/
├── plugin.yaml                    # 插件元数据
├── capabilities/                  # 每个 capability 一个 schema 文件
│   ├── stock_query.json
│   ├── requisition_draft.json
│   └── ...
├── src/
│   ├── __init__.py
│   ├── handlers.py                # capability 入口（业务编排）
│   ├── mapping.py                 # 业务字段 ↔ 外部系统字段
│   ├── client.py                  # 外部系统客户端（HTTP/gRPC/DB）
│   ├── idempotency.py             # 幂等键计算与查表
│   ├── errors.py                  # 错误码翻译表
│   └── config.py                  # 配置 schema（pydantic）
├── tests/
│   ├── contract/                  # capability schema 一致性
│   ├── integration/               # 真实/录制对接
│   └── fixtures/
├── README.md
└── CHANGELOG.md
```

### 3.2 `plugin.yaml`

```yaml
plugin:
  name: mfg_erp_spare
  version: 1.0.0
  display_name: 制造业 ERP 备件插件
  domain: manufacturing
  maintainer: erp-team@company.com

capabilities:
  - capabilities/stock_query.json
  - capabilities/price_query.json
  - capabilities/requisition_draft.json
  - capabilities/purchase_draft.json
  - capabilities/supplier_lookup.json

config_schema:
  endpoint:        { type: string, required: true }
  auth_ref:        { type: string, required: true, secret: true }
  timeout_ms:      { type: integer, default: 3000 }
  warehouse_scope: { type: array<string>, default: [] }

dependencies:
  python:
    - httpx>=0.27
  external_systems:
    - sap_erp >= 7.5

isolation: shared_process    # shared_process | independent_service | sandboxed
```

> `config_schema` 是租户配置的契约。租户在管理台填的字段必须匹配此 schema。

### 3.3 命名约定

- 插件目录：`<domain>_<system>`，如 `mfg_eam`、`fin_credit`、`med_emr`
- Capability：`<domain>.<entity>.<action>`，如 `erp.spare.stock.query`
- 错误码：`<DOMAIN>_<REASON>`，如 `ERP_STOCK_INSUFFICIENT`

## 4. Capability 契约设计

### 4.1 设计原则

Capability 是 Agent 调度的最小单位，契约必须**业务化、可审计、可被 LLM 理解**。

| 原则 | 含义 |
|------|------|
| 业务化命名 | 输入是"设备编号、故障码"，不是 EAM 内部 GUID |
| 输入最小化 | 只要业务必需字段，不让 Agent 拼一堆系统字段 |
| 输出可引用 | 返回数据带 ID、版本、locator，便于回答中引用 |
| 副作用显式 | `side_effect` 必须如实声明 |
| 风险显式 | `risk_level` 决定治理路径 |
| 幂等显式 | `idempotent` + 写操作必须接幂等键 |

### 4.2 Schema 字段速查

| 字段 | 必填 | 说明 |
|------|------|------|
| `name` | ✓ | 全局唯一，遵循 `<domain>.<entity>.<action>` |
| `version` | ✓ | 语义化版本，破坏性变更必须升 major |
| `description` | ✓ | 一句话描述，会被 Planner 提示词使用 |
| `input_schema` | ✓ | JSON Schema |
| `output_schema` | ✓ | JSON Schema |
| `side_effect` | ✓ | `read` / `write_draft` / `write` / `irreversible` |
| `idempotent` | ✓ | 写操作非幂等会被拒绝注册 |
| `risk_level` | ✓ | `low` / `medium` / `high` / `critical` |
| `required_scopes` | ✓ | 调用所需权限 scope |
| `timeout_ms` | ✓ | 超过则平台直接超时返回 |
| `tenant_scope` | ✓ | `tenant` / `cross_tenant`（极少用） |
| `autonomy_ceiling` | ✓ | `L1` / `L2` / `L3` / `L4` |
| `cost_hint` | - | `low` / `medium` / `high` / `external_paid` |

### 4.3 输入输出设计要点

**输入**：

- 必填字段最小化，可选字段说清楚默认行为
- 写操作必须接 `idempotency_key`（或可推导出业务幂等键的字段，如 `incident_id`）
- 不要让 Agent 传内部 ID（如数据库主键），用业务编号

**输出**：

- 读操作：每条结果带稳定 ID，便于 Planner 串联多步
- 写操作（草稿）：返回 `draft_id` + `summary` + `preview`，供前端展示与确认
- 不返回原始系统的杂字段；不要的字段就别返
- 错误情况不要塞进 success 响应，统一走异常路径

### 4.4 版本演进规则

| 变更类型 | 例子 | 版本 |
|------|------|------|
| 加可选字段、加输出字段 | 多返回一个 `alternatives` | minor |
| 加必填输入字段 | 必须传 `warehouse_id` | **major** |
| 改字段含义、改类型 | `stock` 从件数改成箱数 | **major** + 改名 |
| 删字段 | 不再返回 `internal_code` | major |
| 修 bug、改文案 | | patch |

破坏性变更**不要原地改**，发新版本，旧版本保留至少一个发布周期。

## 5. 插件内部分层

### 5.1 三层结构（强约束）

```text
handlers.py  ─ capability 入口，业务编排
    ↓
mapping.py   ─ 业务字段 ↔ 外部系统字段
    ↓
client.py    ─ 调外部系统、协议细节
```

| 层 | 职责 | 不允许 |
|----|------|------|
| handler | 校验输入、组装调用、转换输出、抛业务错误 | 直接拼 SQL、直接发 HTTP |
| mapping | 双向字段映射、枚举翻译、单位换算 | 包含业务规则 / 调外部 |
| client | 协议、鉴权、重试、超时、熔断 | 关心业务字段含义 |

### 5.2 为什么必须分层

- **可测试**：handler 用 mock client 单测；mapping 用纯函数测；client 用录制流量测
- **可替换**：换厂商只动 client + mapping；handler 不变
- **责任清晰**：bug 出在哪一层一眼看出
- **跨插件复用**：mapping 可以共享同行业的字段词典

### 5.3 handler 模板

```python
# handlers.py
from agent_platform.plugin_sdk import capability, PluginContext
from .client import ErpClient
from .mapping import to_stock_dto, from_part_no
from .errors import translate
from .idempotency import idempotency_guard


@capability("erp.spare.stock.query")
async def stock_query(payload: dict, ctx: PluginContext) -> dict:
    part_no = payload["part_no"]
    warehouse_id = payload.get("warehouse_id")

    client = ErpClient(ctx.config)
    try:
        raw = await client.get_stock(
            part_no=from_part_no(part_no),
            warehouse_id=warehouse_id,
        )
    except Exception as e:
        raise translate(e)

    return to_stock_dto(raw)


@capability("erp.spare.requisition.draft")
@idempotency_guard(key_fields=["incident_id", "part_no"])
async def requisition_draft(payload: dict, ctx: PluginContext) -> dict:
    ...
```

要点：

- handler 函数体短、职责单一
- 所有外部异常**必须**在 handler 抛出前翻译为业务错误（详见 §8）
- 写操作**必须**用幂等装饰器或显式幂等键
- 不在 handler 里直接打日志，由 SDK 装饰器统一打（避免遗漏 trace_id）

### 5.4 mapping 模板

```python
# mapping.py
def from_part_no(business_part_no: str) -> str:
    # 业务编号 SP-001 → ERP 内部编号 0000SP001
    return business_part_no.replace("-", "").rjust(10, "0")


def to_stock_dto(raw: dict) -> dict:
    return {
        "part_no": raw["materialNo"],          # ERP 字段 → 业务字段
        "name": raw["materialName"],
        "stock": int(raw["qtyAvailable"]),
        "safety_stock": int(raw["minStock"]),
        "locations": [
            {
                "warehouse_id": loc["whId"],
                "shelf": loc["shelfCode"],
                "qty": int(loc["qty"]),
            }
            for loc in raw.get("locations", [])
        ],
        "alternatives": [],  # 由后续 enrich 填充
    }
```

mapping 是**纯函数**：没有副作用、没有 IO、容易单测。

### 5.5 client 模板

```python
# client.py
import httpx
from .config import ErpConfig
from agent_platform.plugin_sdk import retry, circuit_breaker, secrets


class ErpClient:
    def __init__(self, config: ErpConfig):
        self.config = config
        self._token = secrets.resolve(config.auth_ref)  # 平台 SDK 解析密钥
        self._http = httpx.AsyncClient(
            base_url=config.endpoint,
            timeout=config.timeout_ms / 1000,
            headers={"Authorization": f"Bearer {self._token}"},
        )

    @retry(max_attempts=2, retry_on=(httpx.TimeoutException, httpx.NetworkError))
    @circuit_breaker(failure_threshold=5, reset_after_s=60)
    async def get_stock(self, part_no: str, warehouse_id: str | None) -> dict:
        params = {"materialNo": part_no}
        if warehouse_id:
            params["whId"] = warehouse_id
        resp = await self._http.get("/api/v2/stock", params=params)
        resp.raise_for_status()
        return resp.json()
```

要点：

- 密钥**必须**通过 `secrets.resolve(auth_ref)` 取，不允许写在配置里明文
- 重试和熔断用 SDK 装饰器，不要自己撸（保证统一统计与告警）
- 客户端**不感知业务字段含义**，参数名贴外部系统命名

## 6. 鉴权、密钥与租户配置

### 6.1 配置三段式

```text
plugin.yaml.config_schema   ─ 声明这个插件需要哪些配置项
       ↓
tenant 配置（管理台）        ─ 租户填具体值（含密钥引用）
       ↓
PluginContext.config         ─ 运行时由平台注入到 handler
```

### 6.2 密钥处理强约束

- 密钥**绝不**写入 `plugin.yaml` 默认值
- 密钥**绝不**直接出现在租户配置里，只放引用 `secrets/<name>`
- 密钥实际存储在平台 KMS / Vault，按租户隔离
- handler 拿到的 `ctx.config` 里只有 `auth_ref`，client 通过 `secrets.resolve()` 取真值
- 密钥**不允许**出现在日志、Trace、错误信息里（SDK 自动脱敏）

### 6.3 多租户配置

每个租户启用同一插件可以有不同配置：

```yaml
# 租户 A
plugins:
  mfg_erp_spare:
    config:
      endpoint: https://erp.factoryA.com
      auth_ref: secrets/erp_factoryA_token
      warehouse_scope: ["WH-A1", "WH-A2"]

# 租户 B
plugins:
  mfg_erp_spare:
    config:
      endpoint: https://erp.factoryB.com
      auth_ref: secrets/erp_factoryB_token
      warehouse_scope: ["WH-B1"]
```

handler 通过 `ctx.tenant_id` 自动拿到对应租户的配置，**不需要**自己分支判断。

### 6.4 用户身份穿透

很多外部系统要"以用户身份操作"（如 EAM 工单要记录操作人）。约定：

- 平台在 `ctx.user.external_ids[<system>]` 注入对应外部系统的用户 ID
- client 调用时把它放进请求头或 payload
- 找不到对应外部 ID 时**抛业务错误**`USER_NOT_LINKED`，不要用平台账号代调

## 7. 幂等、超时、重试与熔断

### 7.1 幂等（写操作必备）

任何 `side_effect != "read"` 的 capability 都必须幂等。

**幂等键来源**（优先级）：

1. payload 里业务字段拼接（如 `incident_id + part_no`）—— 最稳
2. 调用方传入 `idempotency_key`
3. 平台自动生成 `request_id`（仅 P0 兜底，不推荐）

**幂等表**：

```sql
CREATE TABLE plugin_execution_record (
  id              bigserial primary key,
  tenant_id       varchar(64) not null,
  plugin          varchar(64) not null,
  capability      varchar(128) not null,
  idempotency_key varchar(256) not null,
  status          varchar(32) not null,    -- in_flight / success / failed
  result          jsonb,
  external_ref    varchar(256),
  created_at      timestamptz default now(),
  updated_at      timestamptz default now(),
  UNIQUE (tenant_id, plugin, capability, idempotency_key)
);
```

SDK 提供 `@idempotency_guard(key_fields=[...])` 装饰器：

- 同键请求二次进入 → 直接返回首次结果
- 首次执行中 → 拒绝并提示
- 失败的请求按业务策略决定是否允许重试

### 7.2 超时

- `timeout_ms` 在 capability schema 强制声明
- client 的实际超时**必须 ≤** schema 声明值，给平台留余量
- 超时不要在 handler 里 `try/except` 吞掉，让 SDK 翻译为 `CAPABILITY_TIMEOUT`

### 7.3 重试

- **只读**操作可以默认重试 1-2 次
- **写操作非幂等**绝不能重试
- **写操作幂等**可以重试，但必须用同一幂等键
- 重试只针对网络/超时错误，**不针对**业务错误（4xx）

### 7.4 熔断

- SDK `@circuit_breaker` 装饰器，按 capability 维度统计
- 触发熔断时返回 `PLUGIN_CIRCUIT_OPEN`，平台会自动切备用路径或告警
- 熔断阈值由插件作者初设，平台 SRE 按租户调整

## 8. 错误翻译与可解释性

### 8.1 错误码三层

```text
外部系统原始异常（HTTP 5xx、SQL Error、SDK Exception）
        ↓ client 层捕获
平台标准异常（PluginException 子类）
        ↓ handler 层抛出
统一对外错误码（业务级文案）
```

### 8.2 错误码翻译表

`errors.py`：

```python
from agent_platform.plugin_sdk import PluginException

class StockInsufficient(PluginException):
    code = "ERP_STOCK_INSUFFICIENT"
    user_message = "库存不足，请确认或选择替代件"
    retryable = False

class ErpUnavailable(PluginException):
    code = "ERP_UNAVAILABLE"
    user_message = "ERP 系统暂时不可用，已切换为只读模式"
    retryable = True

def translate(exc: Exception) -> PluginException:
    if isinstance(exc, httpx.HTTPStatusError):
        if exc.response.status_code >= 500:
            return ErpUnavailable()
        if exc.response.status_code == 404:
            return PartNotFound()
    if isinstance(exc, httpx.TimeoutException):
        return ErpUnavailable()
    return UnknownPluginError(cause=str(exc))
```

### 8.3 业务级错误的要素

每个业务错误必须给：

- **code**：稳定字符串，前端可路由展示
- **user_message**：面向最终用户的中文说明，避免暴露内部细节
- **retryable**：是否可重试（影响 Planner 决策）
- **suggested_action**（可选）：建议的下一步（"换替代件 XX-502"）
- **details**（可选）：结构化字段，前端做更细的展示

### 8.4 不允许做的事

- ❌ 把外部系统的英文报错原样抛给用户
- ❌ 把 stack trace、SQL、URL、token 暴露在错误信息里
- ❌ 用 `Exception("xxx 失败")` 这种没语义的异常

## 9. 审计与可观测

### 9.1 审计

插件**不写**审计日志。审计由平台 SDK 在 capability 调用前后自动写入：

```text
- request_id / trace_id / tenant_id / user_id
- capability + version
- input 摘要（敏感字段已脱敏）
- 调用结果 / 错误码
- 耗时 / 重试次数 / 熔断状态
- 外部 ref（如外部工单号）
```

插件如果有"对业务很重要的中间结果"想留痕，通过 SDK：

```python
ctx.audit.note(key="external_workorder_no", value=external_id)
```

不要直接写自己的日志表。

### 9.2 指标

平台自动暴露每个 capability 的 RED 指标：

- Rate（QPS）
- Errors（按 code 分组）
- Duration（P50 / P95 / P99）

插件作者**不需要**自己埋点，但可以加业务自定义指标：

```python
ctx.metrics.counter("erp.spare.alt_recommended_total").inc()
```

### 9.3 Trace

每次 handler 执行自动作为一个 Trace step，无需手动埋点。如需细分：

```python
with ctx.trace.span("erp.fetch_alternatives"):
    alts = await client.get_alternatives(part_no)
```

## 10. 测试要求

### 10.1 测试分层

| 层 | 工具 | 关注点 |
|----|------|------|
| 合约测试 | jsonschema + pytest | input/output schema 一致性、版本兼容 |
| 单元测试 | pytest + mock | mapping 纯函数、handler 编排 |
| 集成测试 | pytest + recorded fixtures (vcr.py / wiremock) | 真实/录制外部调用 |
| 契约对外测试 | sandbox 租户 + 真实接口 | 上线前最后一关 |

### 10.2 合约测试（强制）

```python
# tests/contract/test_stock_query.py
def test_stock_query_input_schema_valid():
    assert validate(input_schema, {"part_no": "SP-001"}) == OK

def test_stock_query_output_matches_schema(mock_erp_client):
    result = stock_query({"part_no": "SP-001"}, ctx)
    assert validate(output_schema, result) == OK

def test_required_fields_missing_raises():
    with pytest.raises(ValidationError):
        stock_query({}, ctx)
```

CI 中必须全绿才允许打包。

### 10.3 集成测试

录制一份真实 ERP 响应作为 fixture，集成测试基于 fixture 跑：

```python
@pytest.mark.integration
async def test_stock_query_end_to_end(vcr_cassette):
    result = await stock_query({"part_no": "SP-001"}, ctx)
    assert result["stock"] == 12
    assert result["locations"][0]["warehouse_id"] == "WH-A1"
```

集成测试**不**直连生产 ERP。需要时连专用沙箱实例。

### 10.4 红线测试

每个写操作 capability 必须有：

- 同幂等键二次调用，外部系统只被调一次
- 外部系统 5xx 时 handler 抛 `*_UNAVAILABLE`
- payload 缺必填字段时抛 `ValidationError`
- 跨租户 payload 试图穿透时抛 `TENANT_ISOLATION_VIOLATION`

## 11. 插件生命周期与发布

### 11.1 状态机

| 状态 | 含义 | 进入条件 |
|------|------|------|
| `draft` | 本地开发 | 初始化 |
| `validated` | 契约/单元/集成测试通过 | CI 全绿 |
| `approved` | 平台管理员审核通过 | 安全扫描 + 文档完整 |
| `released` | 灰度或全量发布 | 灰度规则配好 |
| `disabled` | 紧急下线 | 出现 P0/P1 事件 |
| `archived` | 停止维护 | 无租户使用 + 半年观察 |

### 11.2 发布流程

```text
1. agent-cli plugin validate    # schema + 依赖检查
2. agent-cli plugin test        # 跑 contract + unit + integration
3. agent-cli plugin pack        # 打包
4. 提交到插件仓库
5. 平台扫描（依赖漏洞、敏感词、license）
6. 管理员审核
7. 沙箱租户启用 1-2 周
8. 灰度租户启用
9. 全量发布
```

### 11.3 升级与回滚

- 同一插件可以同时存在多个版本，由租户选择启用版本
- minor / patch：可热升级
- major：必须新版本走完整发布流程，旧版本保留 ≥ 1 个发布周期
- 发现严重问题：管理员一键 `disabled` → 平台立即把租户切到上一个 `released` 版本

### 11.4 与业务包的版本关系

业务包 `package.yaml` 里写的是**插件版本范围**：

```yaml
plugins:
  - name: mfg_erp_spare
    version: ">=1.0.0,<2.0.0"
```

插件 minor 升级业务包不需重发；major 升级业务包必须显式跟进。

## 13. 常见陷阱与红线

### 13.1 设计陷阱

| 陷阱 | 后果 | 正确做法 |
|------|------|------|
| 把业务规则塞进 handler（如金额阈值判断） | 换厂商时业务规则也丢了 | 阈值放业务包 `approval_policy.yaml` |
| 一个 capability 里干太多事（查 + 算 + 写） | 难审计、难重用 | 拆成多个细粒度 capability |
| 输入字段贴外部系统命名（如 `materialNo`） | LLM 难理解、跨厂商无法复用 | 用业务命名（如 `part_no`） |
| 输出包含外部系统的所有字段 | 数据泄露 + Schema 难稳定 | 显式列字段、按需返回 |
| 错误信息里带 token / URL / SQL | 安全事件 | SDK 自动脱敏 + 错误翻译 |

### 13.2 工程陷阱

| 陷阱 | 后果 | 正确做法 |
|------|------|------|
| 自己写重试循环 | 与平台统计不一致 | 用 SDK `@retry` |
| 自己写日志 | 缺 trace_id、不可关联 | 用 `ctx.trace` / `ctx.audit` |
| 把密钥写进配置默认值 | 密钥泄露 | 只接受 `auth_ref` 引用 |
| 在 handler 里 `time.sleep` 或同步 IO | 阻塞 worker | 全程 async |
| 写操作不接幂等键 | 重复下单 | `@idempotency_guard` 强制 |

### 13.3 安全红线（违反直接拒绝注册）

- 直接读写平台核心表
- 调外部系统时绕过 `ctx.config.endpoint`，写死 URL
- 在错误信息或日志中输出明文密钥 / token
- 跨租户访问数据
- 写操作非幂等

## 14. 插件开发者 Checklist

发布前对照打勾：

**契约**

- [ ] 每个 capability 有完整 schema
- [ ] 命名遵循 `<domain>.<entity>.<action>`
- [ ] `side_effect` / `risk_level` / `idempotent` / `required_scopes` 准确
- [ ] 输入输出字段是业务命名，不是外部系统命名

**代码**

- [ ] handler / mapping / client 三层分离
- [ ] mapping 是纯函数
- [ ] client 用 SDK 的 `@retry` `@circuit_breaker` `secrets.resolve`
- [ ] 写操作用 `@idempotency_guard`
- [ ] 全程 async，无阻塞 IO

**错误**

- [ ] 所有外部异常已翻译为业务级错误
- [ ] 错误 code / user_message / retryable 齐备
- [ ] 错误信息不含密钥、URL、stack

**配置**

- [ ] `plugin.yaml` 的 `config_schema` 完整
- [ ] 密钥只通过 `auth_ref` 引用
- [ ] 多租户配置独立

**测试**

- [ ] 合约测试 100% 覆盖 capability
- [ ] 单元测试覆盖 mapping 和 handler 主路径
- [ ] 集成测试有录制 fixture
- [ ] 红线测试（幂等、租户隔离、外部 5xx）齐备

**文档**

- [ ] README 写明功能、依赖、配置、错误码
- [ ] CHANGELOG 记录每次变更
- [ ] 重要错误码与业务包文档同步

**治理**

- [ ] 不直接读写平台核心表
- [ ] 不绕过 PDP / 不调用其他插件
- [ ] 审计 / 指标 / Trace 都走 SDK
- [ ] 安全扫描（依赖漏洞、license）通过

全打勾 → 提交 `agent-cli plugin pack` 进入发布流程。

---

**结语**

插件开发的核心心态：**插件是平台与外部系统之间的"翻译官"，不是业务大脑**。它只关心"怎么和这个系统说话、说话有没有出错"。业务怎么决策、谁能用、什么时候用——那是业务包和平台治理的事。

写得好的插件具备这些特征：

- 换一家厂商，只动 client + mapping
- 业务规则改了，插件不动
- 出了 bug，一眼定位在哪一层
- 删掉它的人能从 README + capability schema 立刻看懂它在干什么

把这四点做到，插件就合格了。
