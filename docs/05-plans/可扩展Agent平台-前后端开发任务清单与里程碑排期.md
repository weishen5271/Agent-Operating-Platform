## 1. 拆解原则

- 以 PRD 中的 P0a / P0b / P1 / P2 里程碑为主轴。
- 每个阶段均拆分为后端、前端、联调验收三类任务。
- 优先保证“统一对话入口 + 插件能力 + 治理底座”先可用，再逐步放开审批、评估和自动化。
- 任务默认按可交付模块拆分，而不是按岗位机械切块，便于直接建迭代。

---

## 2. 团队角色建议

建议最小配置：

| 角色 | 人数建议 | 主要职责 |
|------|------|------|
| 后端工程师 | 2-3 | Runtime、插件框架、权限、审计、数据层 |
| 前端工程师 | 1-2 | 工作台、对话页、管理后台、审批页、看板 |
| 测试 / QA | 1 | 回归、接口验收、安全与关键链路验证 |
| 产品 / 解决方案 | 1 | 需求澄清、验收口径、业务包优先级 |
| 运维 / SRE | 0.5-1 | 环境、监控、日志、发布与回滚 |

如果资源更紧，建议至少保证 `2 后端 + 1 前端 + 1 QA 兼职`。

## 3. 总体排期建议

| 阶段 | 周期 | 目标 |
|------|------|------|
| P0a | 第 1-6 周 | 统一对话入口、基础编排、检索、插件调用可用 |
| P0b | 第 7-9 周 | 安全、权限、审计、配额、运行时治理补齐 |
| P1 | 第 10-14 周 | 行业包扩展、评估运营、灰度发布、插件仓库 |
| P2 | 第 15-20 周 | 审批执行、受控自动化、平台化运营能力 |

建议节奏：

- 每 2 周一个迭代
- 每个阶段最后 1 周预留联调、修复、验收
- 高风险模块优先做“窄范围 MVP”，避免一开始铺太大

## 4. P0a 基础底座可用（4-6 周）

### 4.1 阶段目标

- 统一工作台与对话页可用
- 支持基础问答、查询、检索增强
- 支持 capability 驱动的插件调用
- 支持最小审计与运行追踪

### 4.2 后端任务清单

#### 4.2.1 基础工程与项目骨架

- 初始化服务工程结构、配置中心、环境配置模板
- 建立 PostgreSQL（含 pgvector）/ Redis 连接封装
- 接入基础日志、Trace ID、请求 ID 中间件
- 建立统一异常码和 API 返回结构

#### 4.2.2 统一对话 Runtime

- 实现 `chat completions` API
- 实现会话、消息、上下文的数据模型
- 实现 Supervisor 请求主链路
- 实现流式响应能力（SSE 或 WebSocket）
- 实现基础响应组装器，支持文本、引用、动作卡片三种输出结构

#### 4.2.3 意图、规划、执行基础能力

- 实现基础 Intent 分类接口与默认分类器
- 实现 Planner 策略路由骨架
- 实现单步直答与基础 `plan_execute` 两种策略
- 实现步骤执行器与统一超时控制
- 实现基础失败归因结构

#### 4.2.4 检索与知识基础能力

- 实现知识文档、知识切片、引用对象模型
- 实现向量检索 + 关键词检索 + 混合召回
- 实现基础重排与引用组装
- 实现知识源最小入库脚本

#### 4.2.5 插件与 capability 基础框架

- 实现 capability schema 结构与校验器
- 实现插件注册中心与启停状态
- 实现插件调用适配层
- 实现内置示例插件：`KnowledgePlugin`、`HRPlugin` 最小读接口
- 实现 capability 输入 / 输出 schema 校验

#### 4.2.6 记忆基础能力

- 实现短期会话记忆写入与读取
- 实现长期记忆最小表结构与读取接口
- 实现上下文摘要拼装逻辑
- 实现记忆写入白名单规则第一版

#### 4.2.7 审计与观测基础能力

- 实现请求级 Trace 主表与步骤表
- 记录意图、规划、检索、插件、模型节点摘要
- 接入 Prometheus 指标：QPS、时延、错误率、Token 消耗
- 建立基础日志检索字段：tenant、user、trace_id、plugin

### 4.3 前端任务清单

#### 4.3.1 工作台首页

- 搭建整体前端工程与路由结构
- 实现登录后首页框架、导航、租户信息展示
- 实现常用能力入口、最近会话、已启用业务包区域
- 实现首页空态与加载态

#### 4.3.2 统一对话页

- 实现会话列表与新建会话
- 实现输入框、发送、停止生成
- 实现消息流式渲染
- 实现普通回答卡片、引用卡片、数据查询卡片
- 实现右侧上下文面板第一版

#### 4.3.3 结果详情页

- 实现复杂结果详情页布局
- 展示结论摘要、引用列表、执行链路摘要
- 支持从对话页跳转查看详情

#### 4.3.4 通用状态体验

- 实现空态、生成中、成功态、失败态组件
- 实现通用错误提示与重试按钮
- 实现基础点赞 / 点踩入口占位

### 4.4 联调与验收任务

- 联调聊天主链路：提问 -> 检索 / 插件 -> 返回引用
- 联调工作台首页的业务包与最近会话数据
- 核验响应结构能覆盖普通问答、数据查询两类场景
- 完成 P0a 冒烟回归

### 4.5 P0a 交付物

- 可用的统一工作台首页
- 可用的统一对话页
- 至少 2 个可调用 capability
- 基础检索问答链路
- 基础审计与观测面板

## 10. 风险与资源建议

### 10.1 主要风险

- 平台核心链路过早追求“大而全”，导致 P0 延期
- 插件契约不稳定，前后端和插件开发无法并行
- 权限、审计、租户隔离如果后补，返工成本很高
- 评估与灰度能力缺失，会导致 P1 后升级风险迅速放大

### 10.2 建议优先级

必须优先做：

- 对话主链路
- capability 契约
- 租户隔离
- 审计 Trace
- 草稿确认机制

可以后置到 P1 / P2：

- Playground 完整版
- 自动化高级调度
- 合规流程自动化增强

### 10.3 推荐执行建议

1. 每个阶段先冻结接口契约，再做页面和服务并行开发。
2. 每个阶段至少保留 20% 时间做联调与修复，不要排满纯开发工时。
3. 对“审批、写操作、跨租户、敏感数据”四类链路单独建高优先级测试集。
